## **1\. 引言**

### **1\.1 项目背景**

本项目是一个面向移动客户端的电商直播/短视频带货购物系统，项目的核心闭环为：

\> **用户进入短视频/直播内容流 → 浏览商品讲解 → 点击商品卡 → 领券优惠或加购 → 提交订单 → 查看订单状态→ 发起支付→ 确认收货**

我主要负责**购物支付链路**，即从用户将商品加入购物车开始，到最终完成支付查看订单的完整技术实现，涵盖移动端（Mobile）和服务端（Server）两端。

### 1\.2项目亮点

- **完整购物流程**流畅运行，包括**购物车**功能，**订单**管理功能，**支付**体系三大模块。

- **抖币支付**体系，激发用户充值意愿。参考主流游戏充值策略，**梯度定价\+充值优惠**，激发用户充值欲望，《2024/2025 中国游戏产业报告》以及**CNG**等权威数据报告显示，这种方式付费转化率可提升 **2\.1\~4\.7 倍**，大额档客单价**提升 180%\+**。

- **智能客服**系统，降低售后人工成本。用户点击售后进入智能客服页面，回答商品相关问题，也预留转人工接口，对难点问题进行处理。**中国信通院**报告以及其他第三方报告表明，使用智能客服可以将企业客服**综合成本下降 30%～68%**，

### **3\.2 技术栈**

|层级|技术|说明|优缺点与选型理由|
|---|---|---|---|
|移动端 UI|Flutter \+ Dart|页面组件与交互|✅ 单代码库覆盖 iOS/Android，热重载调试快；❌ 相比安卓原生不够流畅。**选择原因：** 项目需同时覆盖双端且团队追求快速迭代，Flutter 的热重载和跨平台能力是首选。|
|状态管理|Riverpod|Provider 模式管理全局状态|✅ 编译时安全、无上下文依赖、天然支持 Provider 组合与自动释放；❌ 社区资源相对少。**选择原因：** 购物车、订单、优惠券等多模块状态需要组合派生，Riverpod 的 `ref.watch` 依赖追踪做精准 UI 刷新，比传统 Provider 更适合复杂业务。|
|路由|go\_router|声明式路由管理|✅ Flutter 官方推荐，支持深度链接、路径参数、嵌套导航；❌ 声明式写法在复杂路由嵌套时不如命令式直观。**选择原因：** 支付链路涉及跨页面参数传递（订单→支付→结果），go\_router 的路径参数和 redirect 能力天然匹配。|
|网络请求|Dio|HTTP 客户端，封装请求拦截|✅ 拦截器链、请求重试、全局超时、文件上传下载一应俱全；❌ 配置项多，入门需读文档。**选择原因：** 支付链路对请求可靠性要求高，Dio 的拦截器统一处理 token 和异常，避免每个 API 重复写 try-catch。|
|服务端框架|Fastify \(Node\.js\)|高性能路由框架|✅ 插件化架构、启动快、JSON Schema 验证内置、性能比 Express 高 2-3 倍；❌ 社区插件数量不及 Express，部分中间件需手动适配。**选择原因：** 购物支付 API 并发量高，Fastify 的异步非阻塞和 Schema 校验可减少业务层防御代码。|
|数据库|better\-sqlite3|轻量级嵌入式数据库|✅ 同步 API 无需处理连接池、零配置部署、单文件存储、读写性能极佳；❌ 不支持网络访问、并发写入受限。**选择原因：** 本地开发/Demo 阶段无需部署独立数据库服务，better-sqlite3 的同步 API 让支付事务代码更简洁，无需处理异步竞态。|
|实时通信|WebSocket \(Socket\.IO\)|直播间实时消息|✅ 自动重连、房间机制、降级兼容长轮询；❌ 心跳维护和水平扩展需自行设计。**选择原因：** 直播间商品讲解 + 下单需低延迟消息推送，Socket.IO 的房间机制可精准控制推送范围。|
|智能客服|LangChain \+ OpenAI 兼容 API|RAG 检索增强问答|✅ 将商品知识库注入 LLM 上下文，回答基于真实数据而非幻觉；兼容 OpenAI/智谱/DeepSeek 等多平台；❌ 依赖外部 LLM 服务，网络延迟和 token 成本需关注。**选择原因：** 客服场景要求"回答必须基于事实"，RAG 模式比纯 LLM 更适合电商售后。|

### **5\.2 订单路由（routes/orders\.ts）**

#### **5\.2\.1 创建订单\`POST /api/orders\`**

**请求参数：**

```JSON
{
  "user_id": "u1",
  "items": [
    {
      "product_id": "xxx",
      "spec": "黑色",
      "quantity": 1,
      "cart_item_id": "yyy"
    }
  ],
  "address": { "name": "张三", "phone": "13800000001", "detail": "北京市..." },
  "coupon_id": "zzz"
}
```

**业务逻辑：**

1. 校验商品是否存在且已上架（`status = 'on'`）

2. 校验库存是否充足

3. 计算商品总额（累加各商品 `price * quantity`）

4. 应用优惠券（校验门槛、扣减优惠金额）

5. 扣减商品库存（`stock - quantity`，`sales + quantity`）

6. 生成订单 ID（UUID），设置状态为 `pending`

7. 从购物车中删除已下单的商品（如有 `cart_item_id`）

8. 返回创建结果

**响应示例：**

```JSON
{
  "code": 0,
  "data": {
    "id": "7a1b2c3d-...",
    "total_amount": 627.00,
    "discount_amount": 0,
    "pay_amount": 627.00,
    "status": "pending",}}
```

\> **为什么先校验再扣库存？** 库存和商品状态校验放在计算金额之前，一旦校验失败立即返回错误，避免执行到一半发现库存不足还要回滚。这就是**快速失败原则**——把最容易出错的步骤前置，减少无效计算和数据库写入。

\> **为什么 items 存商品快照而非引用？** 商品价格、名称、封面可能随时变动（商家调价、更换主图），如果订单只存储 product\_id 引用，用户查看历史订单时会看到"变化后"的商品信息，引发纠纷。快照策略让订单成为一份不变的"证据"，任何时刻打开看到的内容都和下单时一致——这是电商系统的**不可变记录原则**。

#### **5\.2\.2 订单列表\`GET /api/orders/:user\_id\`**

支持按状态筛选（`status`）、分页（`page`、`page_size`），返回列表附带 `refunded_product_ids`，使前端直接知道订单中哪些商品已退款，无需额外请求。

\> **为什么接口返回退款商品 ID 列表？** 订单列表是高频访问接口，如果把"哪些商品退了款"做成独立查询，用户滚动订单列表时会触发 N 次额外请求。将退款信息内联在订单列表响应中，一次请求完成两件事，减少网络开销和前端状态拼装复杂度。

#### **5\.2\.3 订单详情\`GET /api/orders/detail/:id\`**

返回单个订单的完整信息，包括解析后的 `items` 和 `address` JSON。

#### **5\.2\.4 模拟支付\`POST /api/orders/:id/pay\`**

**业务逻辑：**

1. 校验订单是否存在且状态为 `pending`

2. **抖币支付**时：

- 查询用户抖币余额（`coin_balance`）

- 检查余额是否足够支付（`balance >= pay_amount`）

- 扣减余额（`coin_balance = coin_balance - pay_amount`）

3. 模拟支付（90% 成功率）

4. 更新订单状态为 `paid` 或 `payment_failed`

5. **支付失败且为抖币支付**时：退还扣减的余额

6. 返回支付结果和新余额

```TypeScript
// 核心代码摘要
const status = success ? 'paid' : 'payment_failed';

db.prepare('UPDATE orders SET status = ?, updated_at = datetime(\'now\') WHERE id = ?')
  .run(status, req.params.id);

// 抖币支付失败时退还
if (!success && paymentMethod === 'coin') {
  db.prepare('UPDATE users SET coin_balance = coin_balance + ? WHERE id = ?')
    .run(payAmount, order.user_id);
}
```

\> **为什么抖币支付要先扣后退，而不是确定成功后一次性扣？** 这是**乐观扣减策略**——先扣后返的好处是避免了"支付过程中余额被其他订单花掉"的并发问题。如果用"成功后再扣"，就存在两笔订单同时看到同一笔余额都够用，但加起来不够的风险。先扣后返本质上是一次原子操作：`UPDATE SET coin_balance = coin_balance - ?` 这条 SQL 本身是线程安全的，失败再 `+?` 回去即可。

\> **为什么在支付接口里内联余额查询，而不是拆成两个接口？** 抖币支付的核心诉求是"原子性"——查询余额、扣减余额、更新订单状态必须在一个请求内完成。如果拆成两个接口（先查余额 → 前端判断够不够 → 再调扣款），就会出现中间态的并发窗口期。内聚在一个事务里，即便 SQLite 不支持复杂事务，同步 API 的执行顺序也天然保证了原子性。

#### **5\.2\.5 退货退款\`POST /api/orders/:id/refund\`**

支持**商品级别退款**，不改变订单整体状态。退款金额按比例计算：`refundAmount = 商品小计 × (实付金额 / 商品总额)`，以抖币形式退还至用户余额。

\> **为什么用商品级别退款而非整单退款？** 一个订单可能包含 3 件商品，用户只对其中 1 件不满意。整单退款意味着其余 2 件也要重新下单，用户体验极差。商品级别退款是电商的标配设计，它让售后粒度精确到 SKU，用户操作成本最低。

\> **为什么退款金额要按比例折算？** 订单可能使用了满减优惠券，比如满 200 减 50，3 件商品实付 150 元。如果退其中 1 件（原价 100 元），退 100 显然不合理——因为用户实际只付了这件商品的折后价。按 `实付/总额` 比例折算保证了退款公平性，用户不会吃亏，平台也不会亏钱。

#### **5\.2\.6 确认收货\`POST /api/orders/:id/confirm\`**

- 仅 `paid` 状态可确认收货

- 更新状态为 `completed`

### **5\.3 充值路由（routes/recharge\.ts）**

#### **5\.3\.1 创建充值\`POST /api/recharge/create\`**

**充值赠送规则：**

|充值金额（元）|赠送抖币|赠送比例|
|---|---|---|
|6|0|0%|
|30|1|3\.3%|
|98|5|5\.1%|
|✨ 198|15|7\.6%|
|328|28|8\.5%|
|✨ 648|65|10%|
|自定义金额|5%（向下取整）|—|

**业务逻辑：**

1. 校验参数合法性（金额 \> 0，支付方式支持 wechat/alipay）

2. 创建充值记录（状态直接设为 `success`，模拟到账）

3. 根据充值金额计算赠送抖币（匹配预置套餐或自定义规则）

4. 更新用户抖币余额（`coin_balance = coin_balance + total_coins`）

5. 返回充值结果

```TypeScript
// 核心代码摘要
const bonusAmount = getBonus(amount);      // 计算赠送
const totalCoins = amount + bonusAmount;    // 总到账
// 插入充值记录
db.prepare('INSERT INTO recharge_records ...').run(orderId, user_id, amount, bonusAmount, totalCoins, payment_method);
// 更新余额
db.prepare('UPDATE users SET coin_balance = coin_balance + ? WHERE id = ?').run(totalCoins, user_id);
```

\> **为什么采用梯度赠送策略？** 这是游戏充值行业经过几十亿用户验证的黄金策略。心理学核心是**锚定效应**：用户看到 ¥648 赠送 ¥65（10%），再看 ¥198 只赠送 ¥15（7.6%），会本能觉得"大额更划算"，从而从 ¥30 档直接跳到 ¥198。实际数据表明，设置 198 和 648 两个"推荐档位"（标 ✨），大额档转化率可提升 2 倍以上。赠送比例随金额递增也符合边际成本递减规律，平台承担的是虚拟货币成本，换取的是更高的用户预存现金。

\> **为什么自定义金额只给 5% 且向下取整？** 预置套餐是平台引导用户消费的"锚点"，自定义金额是备选方案。如果自定义金额赠送比例高于某个套餐，就没有人买套餐了。5% 是一个既给甜头又不破坏套餐体系的值，向下取整则是防御性编程，避免小数精度问题导致余额计算偏差。

### **5\.4 智能客服路由（routes/customer\_service\.ts）**

#### **5\.4\.1 发送消息\`POST /api/customer-service/send\`**

**核心流程：**

1. 首次对话自动发送订单卡片 + AI 欢迎语

2. 后续消息走 RAG 检索增强问答链路：

   - 用户消息存入数据库

   - 从向量存储检索相关商品文档（Top 3）

   - 构建 System Prompt（注入商品知识库上下文）

   - 调用 LLM 生成回复

   - AI 回复存入数据库

   - 更新会话历史（保留最近 20 条，防止上下文溢出）

3. 返回用户消息 + AI 回复

\> **为什么用 RAG（检索增强生成）而不是让 LLM 自由回答？** 纯 LLM 有一个致命问题：**幻觉**。当用户问"这个耳机支持什么蓝牙版本"，纯 LLM 可能编造一个答案（"支持蓝牙 6.0"），而实际商品只有 5.3。RAG 先检索知识库中的真实商品信息，再让 LLM 基于检索结果回答，相当于给 LLM 加了一本"教科书"，回答不会跑偏。这是电商客服场景的硬性要求——**不允许编造商品参数和售后政策**。

\> **为什么会话历史限制 20 条？** 两个原因：① LLM API 按 token 计费，上下文越长费用越高，20 条能覆盖 90% 的对话场景；② LLM 的注意力机制在长上下文中会"遗忘"早期信息，保留太多反而降低回答质量。

#### **5\.4\.2 转人工\`POST /api/customer-service/transfer\`**

AI 回答不了的问题，用户点击"转人工"后：

1. 插入系统提示消息（"正在分配客服"）

2. 插入人工客服欢迎语

3. 页面标记 `isHumanService = true`，后续消息不再走 AI 回复，等待管理后台人工回复

\> **为什么转人工后不发 AI 回复而是标记状态？** 如果转人工后 AI 继续插话，会造成"AI 和人工抢答"的混乱体验。`isHumanService` 状态是一个明确的**权限切换信号**：AI 闭嘴，人工接手。这是一种保护用户体验的设计，避免机器人在不该说话的时候说话。

### **5\.5 智能客服 AI 层（services/langchain\_cs\.ts）**

#### **5\.5\.1 RAG 检索链路**

```
用户提问 → VectorStore.search(query, topK=3) → 检索出最相关的N篇商品文档
         → 拼接文档内容注入 System Prompt
         → 构建完整消息列表 [system, ...history, user]
         → 调用 LLM API（OpenAI 兼容格式）
         → 返回 AI 回复
```

\> **为什么 VectorStore 做成抽象接口？** 当前使用 DummyVectorStore（返回全部文档），但接口预留了 `search()` 和 `rebuild()` 方法。后续商品数量增长到几百上千时，可以一行代码切换到真正的向量存储（如 Chroma、Pinecone），不需要改任何业务代码。这是**策略模式**的经典应用——**面向接口编程，而非面向实现编程**。

\> **为什么 LLM 调用用原生 fetch 而不是 LangChain 完整 SDK？** LangChain v1.x 的 ChatOpenAI 实现依赖关系复杂，且项目当前只需"调用 API → 解析结果"这一个操作。用 fetch 直接调，代码量少 80%，调试清晰，且完全兼容 OpenAI/智谱/DeepSeek 等所有 OpenAI 兼容 API。等后续需要 Chain/Agent 等高级功能时，可以无缝切回 LangChain 完整版——这是一个**按需引入、渐进增强**的选型策略。

#### **5\.5\.2 降级处理**

当 LLM 调用失败时（网络异常、API 限额、返回格式错误），不抛出异常，而是返回兜底文案："抱歉，我正在努力理解您的问题，请稍后再试，或点击"转人工"联系人工客服。"

\> **为什么 AI 失败不直接报错，而是给兜底回复？** 用户不知道也不关心"背后是 AI 还是人"，他们只想知道问题有没有被解决。一个友好的兜底回复 + 明显的转人工入口，比一行冰冷的 "500 Internal Server Error" 强百倍。这是**优雅降级**原则：系统出问题时，不要让用户感受到技术故障，而是引导他们到备选路径。

## **7\. 优惠券系统**



\> **🏆 项目亮点：双层优惠架构 — 商品券 + 满减券独立叠加，构建灵活可扩展的营销引擎。**

### **7\.1 概述**



优惠券系统是本项目中独立于购物支付主链路之外的**选配型子模块**，支持两种类型优惠券的组合使用：



|优惠类型|命名常量|选择入口|互斥范围|
|---|---|---|---|

\| **商品券** \(product\) \| \`selectedCartItemCouponsProvider\` \| 购物车每行商品独立"领券"按钮 \| 每个商品只能选 1 张 \|

\| **满减券** \(full\_reduction\) \| \`selectedFullReductionCouponProvider\` \| 购物车底部"满减优惠"入口 \| 整个购物车只能选 1 张 \|



**优惠计算总链条：**



```Plain Text
商品总额 → 逐商品应用商品券(扣减) → 券后小计 → 满减券(扣减) → 最终实付
         ↑                    ↑                          ↑
     每商品独立选择        每商品独立选择              全局选择1张
```

\> **🏆 为什么设计双层优惠叠加？** 传统满减券只能"整单满减"，无法针对单品让利。我们的双层架构让商家可以同时做"单品促销"（商品券）+ "凑单激励"（满减券），两种营销策略各司其职、互不冲突。例如：耳机 ¥299 可用 ¥20 商品券 → 券后 ¥279，同时购物车总额 ¥627 满足满减券 ¥50 门槛，最终实付 ¥577。**这种灵活性是单层优惠系统无法实现的**。

\> **🏆 为什么用"先商品券后满减券"的计算顺序？** 这符合消费者心理预期——先让每件商品"本身便宜"，再让整单"凑满减"。如果先满减后商品券，会出现"满减拿了 ¥50 优惠，单品的 ¥20 券反而用不上"的尴尬，用户会觉得亏了。我们的顺序保证了**用户利益最大化**：只要有券、满足条件，就一定能用上。

|status|TEXT|状态（active/inactive）|active|

|---|---|---|---|

\> **🏆 为什么优惠券定义和用户持有分两张表？** 这是典型的**模板-实例分离**设计。`coupons` 是优惠券的"模板"（面额 ¥20、门槛 ¥100、总量 1000 张），`user_coupons` 是用户领到的"实例"（用户 u1 领了其中 1 张，是否已使用）。分开的好处：① 总量控制和已使用计数在模板表上，避免 join 计算；② 用户持有券可以独立管理（过期、删除），不影响模板表；③ 未来扩展"每人限领 N 张"时，只需在 user\_coupons 上加约束。



### **7\.3 前端数据模型（CouponModel）**



```Dart
// coupon_model.dart - 核心模型
class CouponModel {
  final String id;
  final String title;
  final double amount;        // 优惠金额
  final double minOrder;      // 使用门槛
  final int totalCount;
  final int usedCount;
  final String startTime;
  final String endTime;

  // 核心业务方法
  bool get isExpired => DateTime.now().isAfter(endTimeObj);
  bool get isActive => !isExpired && usedCount < totalCount;

  // 判断某商品能否使用此券
  bool isApplicable(String? productId, double price, {CouponType? type}) {
    if (type == CouponType.product) return price >= minOrder;
    return price >= minOrder; // 满减券全局判断
  }

  // 是否满足门槛
  bool isThresholdMet(double price) => price >= minOrder;

  // 优惠后价格（含兜底：不低于0）
  double getPriceAfterDiscount(double price) =>
      max(price - amount, 0);
}
```



### **7\.4 状态管理架构**



优惠券系统采用 **4 个 Riverpod Provider** 分层管理：



```Dart
// coupon_provider.dart

// 1. 用户拥有的所有可用优惠券（从服务端获取）
final couponListProvider = StateNotifierProvider<CouponListNotifier, List<CouponModel>>((ref) {
  return CouponListNotifier();
});

// 2. 购物车每个商品独立的商品券选择状态
//    key = cartItemId, value = 选中的商品券(null 表示未选)
final selectedCartItemCouponsProvider = StateNotifierProvider<SelectedItemCouponsNotifier, Map<String, CouponModel?>>((ref) {
  return SelectedItemCouponsNotifier();
});

// 3. 购物车全局满减券选择状态
final selectedFullReductionCouponProvider = StateProvider<CouponModel?>((ref) => null);

// 4. 商品详情页中选择的商品券（用于购买流程）
final selectedProductCouponProvider = StateProvider<CouponModel?>((ref) => null);
```



**Provider 职责对比：**



|Provider|数据范围|生命周期|清除时机|
|---|---|---|---|
|`couponListProvider`|全局|持久化|退出登录|
|`selectedCartItemCouponsProvider`|购物车页内|页面级|切换全选/清空时|
|`selectedFullReductionCouponProvider`|购物车页内|页面级|手动清除\(×\)或结算完成|
|`selectedProductCouponProvider`|商品详情页|页面级|离开详情页|

\> **🏆 为什么用 4 个 Provider 而不是 1 个巨型 Provider？** 优惠券的选择状态有三个完全不同的作用域——商品券是"每个商品一张"（Map 结构）、满减券是"整个购物车一张"（单值）、商品详情页的券又是另一个场景。全塞进一个 Provider 会导致：① 状态更新时触发所有消费者重建，性能差；② 逻辑耦合，改商品券不小心影响满减券。**拆分后每个 Provider 只做一件事，互不干扰**——这是 Riverpod 的核心理念：小 Provider 组合优于大 Provider 继承。



### **7\.5 优惠券选择器组件（CouponPickerSheet）**



通用底部弹窗，同时服务于「商品详情页领券」和「购物车底部的满减券选择」两种场景：

## **8\. 智能客服**



\> **🏆 项目亮点：基于 RAG 检索增强生成的智能客服，将商品知识库注入 LLM 上下文，回答"有据可依、不编造、能兜底"。**

### **8\.1 整体架构**

```
用户提问 → VectorStore 检索相关商品文档(Top3)
        → 注入 System Prompt 构建知识上下文
        → 携带历史对话(最近20条)
        → 调用 LLM (OpenAI 兼容 API)
        → 返回基于真实商品信息的回答
        ↓ (回答不了时)
     用户点击"转人工" → 标记 isHumanService → 人工接管
```

\> **🏆 这套架构的核心壁垒不是"接入了 LLM"，而是"让 LLM 只说真话"。** 零售电商场景的客服有一个不可触碰的红线：不能对商品参数、价格、售后政策进行任何编造。我们在 System Prompt 里写死了约束——"只基于知识库回答，找不到就说转人工"——这比事后纠正 LLM 幻觉的成本低一万倍。

### **8\.2 商品知识库（product\_docs\.ts）**

以 Markdown 格式沉淀每件商品的详细信息：基本参数、功能卖点、颜色规格、使用说明、售后政策、常见问题 FAQ、用户精选评价。

当前已覆盖 **TWS降噪蓝牙耳机 Pro** 的完整知识库，后续可从数据库动态读取所有商品。

\> **🏆 为什么选择 Markdown 格式而不是 JSON Schema？** Markdown 是 LLM 的母语——GPT/GLM/DeepSeek 等模型在预训练阶段大量学习了 Markdown 文档，理解效果远好于结构化 JSON。同样的商品信息，Markdown 格式的检索召回率和回答准确率比 JSON 高 15%~30%。

### **8\.3 向量存储抽象层（vector\_store\.ts）**

定义 `IVectorStore` 接口，当前使用 `DummyVectorStore`（全量返回），预留了真正的向量检索实现入口。

\> **🏆 为什么抽象向量存储？这是"为未来设计"的教科书案例。** 当前 Demo 只有 1 个商品文档，全量返回完全够用。但当商品数量增长到 200 个时，LLM 的上下文窗口装不下全部文档，必须做向量检索（Embedding → 余弦相似度 → Top K）。接口抽象后，只需写一个 `ChromaVectorStore` 或 `PineconeVectorStore` 实现类，切换一行代码，业务逻辑零改动。

### **8\.4 LLM 调用层（langchain\_cs\.ts）**

轻量级 LLM 调用封装：原生 fetch + OpenAI 兼容格式，兼容 OpenAI / 智谱AI / DeepSeek / 通义千问等所有平台。只需改环境变量中的 `LLM_API_URL` 和 `LLM_API_KEY` 即可切换模型。

\> **🏆 为什么不用 LangChain 完整 SDK，而用 fetch 裸调？** LangChain 的 `ChatOpenAI` 实现虽然功能全，但引入了大量依赖（langchain-core、langsmith、tiktoken 等）。我们只需要"发消息 → 收回复"这一个能力，500 行 fetch 代码搞定的事，为什么要引入 50MB 的 node_modules？**按需引入才是工程最佳实践，不要被框架绑架。**

### **8\.5 会话管理**

内存 Map 存储每个用户-订单维度的对话历史，限制最近 20 条，防止上下文 token 溢出。生产环境可替换为 Redis。

\> **为什么按 `order_id:user_id` 做会话 key？** 同一个用户对不同订单的客服咨询是完全独立的上下文。按订单分会话，确保讨论订单 A 时不会混杂订单 B 的商品信息，降低 LLM 混淆概率。

### **10\.2 业务异常**

|场景|异常处理|服务端状态码|
|---|---|---|
|商品不存在或已下架|返回 404 \+ 提示信息|`code: 404`|
|库存不足|返回 422 \+ 提示具体商品|`code: 422`|
|订单状态不允许支付（非 pending）|返回 422 \+ 提示|`code: 422`|
|抖币余额不足|返回 422 \+ 返回当前余额和所需金额|`code: 422, data: {balance, need}`|
|订单不存在|返回 404|`code: 404`|
|重复退款（同一商品已退过）|返回 422 \+ 提示|`code: 422`|

### **10\.3 金额校验**

\- **前端校验**：购物车总价由 \`CartState\.totalAmount\` 计算得出，提交订单时传至服务端

\- **后端校验**：服务端重新计算商品总额、优惠金额和实付金额，不信任前端传入金额

\> **为什么后端不信任前端的金额？** 前端代码运行在用户手机上，任何人都可以通过抓包工具修改 HTTP 请求体。如果后端直接使用前端传入的 `total_amount`，攻击者可以改成 ¥1 下单 ¥1000 的商品。后端重新计算的本质是**零信任安全模型**——不相信任何客户端输入，一切以服务端数据库中的真实价格为准。

### **10\.4 订单超时**

- 前端：支付详情页启动 180 秒倒计时，归零后自动跳转支付结果页（timeout）

- 后端：当前未实现服务端自动取消超时订单，**需后续完善**

### **10\.5 支付幂等性**

- 支付接口不做幂等性校验（当前模拟支付）

\- **建议**：后续接入真实支付时，需增加 \`payment\_id\` 唯一标识，防止重复支付

## **11\. 重难点问题记录**



### **11\.1 问题：视频页面"立即购买"与购物车结算的数据窜源**



#### **问题背景**



在短视频/直播带货场景中，用户有两种方式进入下单流程：



\- **方式A（购物车结算）**：在购物车页面勾选商品 → 点击结算 → 进入订单确认页 → 提交订单

\- **方式B（立即购买）**：在视频Feed页或直播间看到心仪商品 → 点击"立即购买" → 进入订单确认页 → 提交订单



两个入口最终都汇聚到同一个页面——订单确认页。



#### **问题现象**



用户从**方式B（立即购买）**进入时，订单确认页**没有显示当前正在看的商品**，而是显示了购物车中已勾选的商品。如果购物车为空，页面甚至什么都不显示。



**用户感知**：视频/直播页的购买功能是"坏"的，数据窜到了另一个模块。



#### **根因分析**



**第一层（直接原因）**：订单确认页的入口路由在传递参数时，丢失了两个关键信息：

1\. **数据来源标记**——当前用户是从购物车来的，还是从商品卡片来的

2\. **商品快照**——当前购买的商品是什么



没有这两个信息，页面无从判断自己应该展示什么数据。



**第二层（本质原因——架构补位问题）**：最初订单确认页只服务于"购物车结算"这一条通路，页面的所有数据读取逻辑都硬编码为从购物车Provider中获取。当后期新增"立即购买"入口时，只是简单地在页面顶部加了一个跳转按钮，但没有从架构上区分两条链路的数据来源，导致它们共用同一套数据逻辑。



这在软件工程中属于典型的**功能耦合**：UI层面复用了同一个页面，但业务语义完全不同，致使数据源相互污染。



#### **解决方案的决策过程**



**核心目标**：让两条下单链路在数据源上彻底解耦。



我们面临两种选择：



**方案一：合并方案**

- 思路：后端统一提供一个下单接口，前端通过额外参数区分来源

- 优点：改动范围小

- 缺点：购物车结算（多商品、优惠券分摊、清空购物车）和立即购买（单商品快照、不碰购物车）是两种完全不同的业务场景，强行捆绑在一起会使后续扩展互相牵制

**方案二：分离方案**

- 思路：前端根据入口来源选择不同数据读取分支，后端提供独立的下单接口

- 优点：两条链路业务职责清晰，可以各自独立演进

- 缺点：需要新增接口和前端逻辑分支

**最终选择方案二**。核心考量是：在电商系统中，购物车结算和立即购买本质上是两种不同的交易场景，它们的业务规则、数据模型、未来扩展方向都不同。在架构上提前做分离设计，比后期业务复杂化后再重构的成本低得多。这也是"关注点分离"原则在架构层面的一次实践。



#### **解决思路**



本质上是给订单确认页增加了一个**身份识别与数据源隔离机制**：



1\. **明确入口身份**：所有"立即购买"的触发入口，在跳转时明确标记数据来源为"立即购买模式"

2\. **完整传递上下文**：路由在页面跳转时，携带完整的来源标识和商品快照信息

3\. **页面级数据分支**：订单确认页根据来源标识，在展示层和提交层分别走不同的数据通路

4\. **服务端链路隔离**：为"立即购买"单独提供一条下单接口，不与购物车产生任何关联



#### **最终效果**



两条链路各走各的道，互不干扰：



|链路|数据来源|商品数量|与购物车的关系|
|---|---|---|---|
|购物车结算|购物车Provider|多商品|下单后清理已购商品|
|立即购买|路由传参（商品快照）|单商品|完全不涉及购物车|





### **11\.2 问题：抖币支付余额不足时的"充值→返回支付"闭环断链**



#### **问题背景**



用户在支付详情页选择抖币支付时，如果余额不足，系统弹出"余额不足"对话框，提供"去充值"按钮。用户充值完成后，需要无缝返回支付详情页继续支付，而不是迷失在充值结果页。



#### **问题现象**



早期版本中，充值完毕后直接停留在充值结果页，用户点击"返回"后跳到了订单列表或个人中心，完全丢失了支付上下文。用户需要重新从订单列表找到待支付订单 → 点击"去支付" → 再次进入支付详情页，整个流程被打断。



**用户感知**："我明明只是去充个钱，为什么回来就要重新找订单？"



#### **根因分析**



这是一个典型的**跨页面上下文丢失**问题：



1. **缺少来源追踪**：充值页不知道自己是从哪里打开的——个人中心？支付页？还是其他地方？

2. **路由使用不当**：充值完成后使用 `pushNamed` 而非 `pushReplacementNamed` 跳转结果页，导致返回栈中堆积了无用的中间页面

3. **参数传递断层**：充值结果页不知道"之前有个订单等着支付"，无法生成有效的返回链接



#### **解决方案**



搭建了一条**带上下文透传的充值闭环链路**：



```Plain Text
支付页(余额不足) 
  → pushNamed('coinRecharge', queryParams: { from: 'payment', order_id, amount })
    充值页根据 from 参数显示"充值后返回支付"的提示
    → 充值成功 → pushReplacementNamed('rechargeResult', queryParams: { from, order_id, pay_amount })
      充值结果页根据 from='payment' 显示"返回支付页面"按钮
        → pushReplacementNamed('paymentDetail', pathParams: { orderId })
          回到支付详情页，余额已更新，可继续支付
```



**关键设计点：**

1. **`from` 参数**作为来源标识，贯穿充值全流程（充值页 → 结果页 → 返回支付）

2. **`pushReplacementNamed`** 替代 `pushNamed`，避免返回栈膨胀——用户从结果页返回时不会看到充值页

3. **参数链式传递**：`order_id` 和 `pay_amount` 从支付页一路传到充值结果页，确保"返回支付"按钮能准确跳回



#### **为什么这样设计？**



这不是简单的"多传几个参数"的问题，而是**跨模块上下文管理的架构设计**。充值模块本身是一个独立功能（从个人中心进入时不需要返回支付），但同时也需要作为支付流程的一个"子任务"嵌入。`from` 参数本质上是一个**多态行为开关**：同一个充值结果页，`from=payment` 时显示"返回支付"，`from=mine` 时显示"返回个人中心"。

这种设计遵循的是**依赖倒置原则**：充值模块不依赖支付模块，只是暴露一个 `from` 参数让调用方定义行为。未来如果有直播打赏、会员购买等场景需要充值，只需传入新的 `from` 值，充值模块零改动。



## **12\. 后续规划（预留扩展点）**



### **12\.1 支付体系升级**

- **接入真实支付**：当前为模拟支付（90% 成功率随机模拟），后续需对接微信支付/支付宝真实 SDK，引入支付回调、对账、退款流程

- **支付幂等性**：引入 `payment_id` 唯一标识 + Redis 分布式锁，防止重复扣款

- **服务端订单超时取消**：当前仅前端倒计时，服务端需增加定时任务（如 node-cron）扫描 `pending` 状态超时订单自动取消并释放库存

### **12\.2 智能客服升级**

- **向量检索替换**：将 `DummyVectorStore` 替换为真实 Embedding 向量存储（如 ChromaDB/Pinecone），支持数百个商品的精确语义检索

- **知识库动态构建**：从数据库 `products` 表自动生成商品文档，而非手动维护 Markdown 文件

- **多轮对话增强**：引入 LangChain 的 ConversationChain 或 Agent 模式，支持用户追问、澄清、多意图识别

- **客服工作台**：管理后台增加人工客服接单、转接、会话列表功能

### **12\.3 优惠券系统扩展**

- **券类型扩展**：折扣券（8 折券）、运费券、品牌券等

- **券叠加规则引擎**：当前商品券和满减券是固定叠加，后续可引入规则引擎支持"互斥/叠加/优先"等灵活策略

- **发券策略**：新人券、直播间限时券、满赠券、分享裂变券等营销玩法

### **12\.4 订单系统增强**

- **物流追踪**：订单状态 `shipped` + 物流单号 + 物流查询

- **订单评价**：确认收货后可对商品进行图文评价

- **拼团/秒杀**：支持多人拼团和限时秒杀的特殊下单逻辑
