### B1. 从购物车点"结算"到支付成功，经历了哪些步骤？

**这道题在问什么**
答辩官看到你在 Demo 里点了几下就支付成功了，他想确认你不是只会点按钮，而是能说清楚数据在页面之间怎么流动。

**你需要先理解的概念：Flutter 页面跳转机制**

Flutter 用 `go_router` 管理页面跳转，可以理解为一本"电话簿"——每个页面有一个名字（比如 `paymentDetail`）和对应的路径（比如 `/payment/detail/:orderId`）。跳转时有两种方式：

- **pushNamed**：在"页面栈"上面叠一层新页面。用户点返回键时，会回到上一个页面。适用于"进一扇门，还能退回去"的场景，比如从购物车进入订单确认页。
- **pushReplacementNamed**：把当前页面替换掉。用户点返回时，不会回到被替换的页面。适用于"这道门走过了就不需要回头"的场景，比如支付成功后，就不需要再看到支付详情页了。

在我们的支付链路中，充值→充值结果→返回支付详情，全程用 `pushReplacement`，用户不会在返回时看到一堆中间页面。

**答辩精简回答**（30 秒版）
五步：① 购物车选中商品 → 点结算 → 进入订单确认页（填地址、看费用）；② 提交订单 → 后端创建订单、扣库存；③ 进入支付详情页（180 秒倒计时）；④ 选抖币/微信/支付宝 → 确认支付 → 后端处理扣款；⑤ 跳到支付结果页。全程用 `go_router` 的 `pushReplacementNamed` 保证返回栈干净。

**追问：为什么有些跳转用 pushReplacement？**
充值后回到支付页，如果用 `push`，返回栈是：支付页 → 充值页 → 充值结果页 → 支付页，用户点返回会看到充值结果页和充值页，体验很差。用 `pushReplacement`，充值结果页直接替换支付页，返回按钮直接回到订单列表。

### B2. 订单里的商品信息，为什么不直接引用商品表，而是存了一份快照？

**这道题在问什么**
答辩官在你的文档 2.3 节看到了"items 存商品快照"，他想知道你为什么做这个选择。这背后是软件设计的一个基本原则。

**你需要先理解的概念：不可变记录**

假设一个真实场景：用户今天花 ¥299 买了一个耳机，订单记录了这个价格。一个月后商家搞促销把价格改成了 ¥199。如果订单只存了一个"商品 ID 引用"，用户打开历史订单时，价格显示 ¥199——用户会以为自己被多收了 ¥100，引发纠纷。

"快照"的做法是：下单那一刻，把商品的名称、价格、封面图全部复制一份存进订单。以后不管商品怎么改，订单里的内容永远不变。这就像拍了一张照片——照片里的人老了，但照片不会变。

**答辩精简回答**（30 秒版）
存的是商品快照而非引用，因为商品信息（价格、名称、封面）可能随时被商家修改。如果只存引用，用户查历史订单时会看到变化后的商品信息，引发纠纷。快照让订单成为一份任何时刻打开都一致的记录。

**追问：快照会占用更多存储空间吗？**
会，每个订单多存约 200 字节（商品名+价格+封面 URL），但对于电商系统的法律确定性来说，这点空间成本可以忽略。而且 JSON 字段在 SQLite 中存储效率很高。

### B3. 订单状态有哪些？每次状态变化前做了什么检查？

**这道题在问什么**
答辩官想知道你是否理解"状态机"——订单不是随意切换状态的，每一步都有前置条件。

**你需要先理解的概念：状态机的"守卫"**

状态机就像一个地铁闸机——你要进站，必须先刷卡（前置条件），闸机才会打开（状态变更）。如果你不刷卡硬闯，闸机会报警（返回错误）。

订单也一样：`pending`（待支付）变成 `paid`（已支付）之前，必须检查：① 订单当前确实是 pending 状态；② 如果是抖币支付，余额必须够。如果订单已经是 paid 状态了，再点支付就会返回"订单状态不允许支付"。

**答辩精简回答**（30 秒版）
四个核心状态：pending（待支付）→ paid（已支付）→ completed（已完成）。每次转换都有前置检查：支付要验证 status=pending 且余额够；确认收货要验证 status=paid。还有一个 payment_failed 状态，抖币支付失败时余额会退还。

**追问：退款后的订单状态会变吗？**
不会。退款是商品级别的，不改变订单整体状态。一笔订单有 3 件商品，退了 1 件，订单仍然是 completed，只是那 1 件商品的钱按比例退还了。退款记录单独存在 refund_records 表里。

### B4. 抖币支付如果失败，怎么保证用户余额不丢？

**这道题在问什么**
这是并发安全的核心问题。答辩官想知道你考虑了"两个请求同时操作同一笔余额"的风险。

**你需要先理解的概念：先扣后退 + 并发窗口**

想象一个场景：用户余额 ¥100，同时下了两笔订单，每笔 ¥80。如果流程是"先检查余额 → 都够 → 再扣款"，两笔订单同时检查时都看到 ¥100 够用，结果各扣 ¥80，余额变成 -¥60——用户倒欠平台钱了。

我们的做法是**先扣后退**：① 先把余额扣掉（`余额 = 余额 - 80`），这条 SQL 语句在数据库里是排队执行的，两笔订单不可能同时扣同一笔余额；② 执行支付逻辑；③ 如果支付失败，再把扣掉的钱加回去。SQLite 的同步 API 保证了这个过程的顺序性。

**答辩精简回答**（30 秒版）
采用"先扣后退"策略：支付前先扣余额，支付成功则确认，失败则退还。这样避免了"查余额和扣余额之间的并发窗口"——两个请求同时看到余额够用但加起来不够的问题。SQLite 的同步 UPDATE 语句天然排队执行。

**追问：如果服务器在扣款后、退款前崩溃了怎么办？**
这是进程崩溃问题。当前 SQLite 单文件 + WAL 模式提供了 Crash-Safe 保证——事务要么完整提交，要么完全回滚。极端情况下如果余额扣了但订单状态没更新，可以通过数据库日志恢复。但最佳的解决方案是引入分布式事务或 Saga 模式——这是后续接入真实支付时要补的。

### B5. 退款金额为什么不是退商品原价？你是怎么算的？

**这道题在问什么**
答辩官看到文档 7.3 节有一个公式，想知道你理解这个公式的业务含义。

**你需要先理解的概念：按比例折算**

场景：用户买了 3 件商品共计 ¥200，用了一张"满 200 减 50"的优惠券，实付 ¥150。后来退其中 1 件原价 ¥100 的商品。

如果退 ¥100：用户实际只付了这件商品折后价（约 ¥75），却拿回了 ¥100，相当于白赚 ¥25——平台亏了。

正确做法：按比例折算。这件商品占总金额的比例是 100/200 = 50%，实付 150 的 50% 是 75 元。退 ¥75，用户不亏、平台不亏。

**答辩精简回答**（30 秒版）
公式是 `退款金额 = 商品小计 × (实付金额 / 商品总额)`。因为订单可能用了满减券，实付金额低于商品总额，如果按原价退等于平台倒贴优惠券差价。按实付比例折算保证双方公平。

### B6. 充值赠送规则是怎么设计的？为什么搞阶梯赠送？

**这道题在问什么**
答辩官在你的文档 3.1 节看到"参考游戏充值策略"和锚定效应，他想确认你不是随便定的比例，而是有业务逻辑支撑的。

**你需要先理解的概念：锚定效应**

心理学实验证明：人做决策时会过度依赖第一个看到的信息（"锚"）。在充值场景中：用户先看到 ¥648 赠送 ¥65（10%），再看 ¥198 赠送 ¥15（7.6%），大脑会本能判断"大额更划算"——¥198 本身不差，但在 ¥648 的对比下显得"亏了"。

游戏行业几十年来用这个策略让玩家从首充 ¥6 一路跳到 ¥648，数据证明付费转化率能提升 2 倍以上。

**答辩精简回答**（30 秒版）
参考游戏行业的锚定效应：¥648 送 10%（65 抖币），¥198 送 7.6%（15 抖币），阶梯差让用户本能觉得大额更划算。预置套餐固定赠送，自定义金额统一 5%。数据支撑来自《中国游戏产业报告》——梯度定价使付费转化率提升 2.1~4.7 倍。

### B7. 智能客服怎么保证不说假话？万一问了知识库没有的商品呢？

**这道题在问什么**
答辩官可能用过 ChatGPT，知道 AI 会"一本正经地胡说八道"。他关心你有没有防范措施。

**你需要先理解的概念：RAG（检索增强生成）**

普通 LLM 就像一个博学但爱编造的朋友——你问他"耳机蓝牙版本"，他可能随口说个"5.4"，实际上你的耳机是 5.3。因为他没有你的商品资料，只能凭记忆编。

RAG 的做法是：在 LLM 回答之前，先给他塞一份"小抄"——从你的商品知识库里搜出最相关的 3 篇文档，告诉他"只能参考这个回答，找不到就说不知道"。这样 LLM 的回复就有了事实依据。

System Prompt 是写给 LLM 的"行为准则"，里面明确写了："如果知识库中找不到，就说转人工"。这个边界不是用代码硬编码的，而是靠 Prompt 约束——好处是加了新商品不需要改代码。

**答辩精简回答**（30 秒版）
用了 RAG（检索增强生成）：用户提问后，系统先从商品知识库检索最相关的文档，注入到 System Prompt 里告诉 LLM"只能基于这个回答，找不到就说转人工"。这样 LLM 不会编造商品参数。问知识库外的商品，LLM 会引导用户转人工。

**追问：如果 LLM 不听话，无视 System Prompt 编造了答案怎么办？**
那问题可能出在：① Prompt 写得不够强硬——可以加"你绝对不能编造任何商品信息，否则会造成严重的商业后果"；② 模型温度（temperature）太高——降到 0.3 以下会显著减少随机编造；③ 终极方案是加一层后处理过滤——检测回复中是否包含不在知识库中的参数值，如果包含则自动替换为转人工提示。

### B8. "立即购买"和"购物车结算"有什么区别？为什么分开做？

**这道题在问什么**
答辩官在 Demo 里可能看到你既从购物车下单、又从视频页直接点"立即购买"，两个入口到了同一个确认页。他想知道这背后有没有数据混淆的风险。

**你需要先理解的概念：数据源隔离**

购物车结算 = 从购物车 Provider 里拿数据（多商品、用了优惠券、下单后要清购物车）。
立即购买 = 从视频页的路由参数里拿数据（单商品快照、不经过购物车、不碰购物车表）。

如果两个入口共用一套逻辑，会出现"立即购买时显示了购物车里的商品"这种 bug——这就是"功能耦合"。分离后每条链路独立，互不干扰。

**答辩精简回答**（30 秒版）
购物车结算是多商品 + 优惠券分摊 + 下单后清购物车；立即购买是单商品快照，不经过购物车。最初两个入口共用一页导致数据窜源，后来做了分离：前端根据 `from` 参数走不同分支，后端提供两套独立接口（`POST /api/orders` 和 `POST /api/orders/direct`）。

**追问：为什么选择分离方案而不是在前端加一个参数区分？**
购物车结算（多商品、优惠分摊、清空购物车）和立即购买（单商品快照、不碰购物车）是两种完全不同的交易场景。合并方案短期内改动小，但随着业务复杂化——比如购物车加拼团、立即购买加闪购——捆绑在一起的代码会互相牵制。提前分离是"关注点分离"原则的实践，长期维护成本更低。

## 二、项目亮点考察（10 题）

### H1. 为什么做抖币支付体系，而不是直接对接微信/支付宝？

**这道题在问什么**
答辩官可能觉得"直接对接微信支付不就完了"。他想听你从商业角度解释虚拟货币的价值。

**你需要先理解的概念：平台货币 vs 第三方支付**

对接微信支付：每笔交易 0.6% 手续费，资金直接到用户银行卡，平台一分钱不留。
做抖币体系：用户先充值到平台（预存现金），再用抖币消费。平台沉淀了大量用户资金（就像星巴克的星礼卡），还能通过赠送给用户"占便宜"的感觉——赠送的抖币边际成本为零。

**答辩精简回答**（30 秒版）
微信/支付宝每次支付有手续费，资金也不留在平台。抖币体系让用户先充值再消费，资金沉淀在平台，赠送的抖币边际成本为零，同时梯度赠送制造"多充多送"的消费心理。参考游戏行业数据，这种模式付费转化率可提升 2 倍以上。

**追问：抖币支付有合规风险吗？**
需要关注虚拟货币的监管政策。当前设计下抖币是单向充值（人民币→抖币）、不支持提现（抖币→人民币），这在法律上属于"预付费虚拟道具"而非金融产品，合规门槛较低。如果未来加提现功能，就需要支付牌照。

### H2. RAG 检索 + LLM 是怎么配合的？和直接调用 ChatGPT 有什么不同？

**这道题在问什么**
这是文档第 4 章的核心亮点。答辩官想确认你理解 RAG 和普通 LLM 的本质区别。

**你需要先理解的概念：Embedding 向量检索**

想象你要在图书馆找一本讲"蓝牙耳机降噪"的书。传统方式是翻目录（关键词匹配），但"降噪"和"隔音"是两个词，关键词匹配找不到。

向量检索的做法：把每本书的内容转成一串数字（向量），"降噪"和"隔音"的数字串很相似。用户提问也转成向量，和所有书的向量做"余弦相似度"计算，找出最相关的 3 本。这个过程叫 Embedding。

我们当前用 DummyVectorStore（全量返回所有文档），因为只有 1 个商品的文档。但接口已经抽象好了，商品多了之后换 ChromaDB/Pinecone 一行代码搞定。

**答辩精简回答**（30 秒版）
直接调 ChatGPT 可能编造商品参数（"这个耳机支持蓝牙 6.0"实际上只有 5.3）。RAG 先检索知识库中的真实商品文档，注入到 System Prompt 约束 LLM 只能基于检索结果回答。VectorStore 做了接口抽象，当前全量返回，后续可无缝切换向量数据库。

### H3. 双层优惠券为什么先商品券后满减券？反过来会怎样？

**答辩精简回答**（30 秒版）
先满减后商品券，满减已经扣了 ¥50，整单金额降低，某些商品券的门槛可能就达不到——用户虽然有券却用不上。先商品券后满减券保证用户利益最大化，每张符合条件的券都能用上。天猫/京东也是先单品优惠后平台优惠的顺序。

**追问：如果业务要求"某些券互斥、某些叠加"怎么办？**
当前是固定叠加，后续可引入一个轻量级规则引擎：`rules = [{ type: 'product', exclusiveWith: ['fullReduction'] }]`。每种券标记互斥组，计算前先检查已使用的券列表中是否有互斥券。

### H4. VectorStore 做成接口是出于什么考虑？这体现了什么设计原则？

**你需要先理解的概念：策略模式**

策略模式的核心思想是"把容易变化的部分封装起来"。你有一个"搜索"的需求，但搜索的具体方式可能变化——现在全量返回（1 个文档够用），未来需要向量检索（200 个文档装不下）。

做法：定义一个接口 `IVectorStore`，里面有一个 `search()` 方法。当前用 `DummyVectorStore` 实现（全量返回），未来写一个 `ChromaVectorStore` 实现（向量检索），改动只发生在工厂函数那一行，所有调用 `search()` 的代码完全不需要改。

**答辩精简回答**（30 秒版）
体现了策略模式。定义 `IVectorStore` 接口，当前 DummyVectorStore 全量返回（1 个文档够用），后续只需写一个 ChromaVectorStore 实现类，工厂函数里改一行，业务代码零改动。核心原则：面向接口编程，不面向实现编程。

### H5. 充值模块的 `from` 参数是什么设计思想？如果加新场景需要改充值代码吗？

**你需要先理解的概念：依赖倒置**

传统做法：充值模块内部写死"充值完后跳到支付页"——这叫充值模块依赖支付模块。问题是未来直播打赏也需要充值，充值模块就得改代码。

依赖倒置的做法：充值模块不关心"谁在调用我"，只暴露一个 `from` 参数。调用方告诉我 `from='payment'`，我就显示"返回支付"按钮；告诉我 `from='mine'`，我就显示"返回个人中心"。新场景只需传入新的 `from` 值，充值核心逻辑零改动。

**答辩精简回答**（30 秒版）
体现了依赖倒置原则——充值模块不依赖任何调用方，只通过 `from` 参数暴露行为接口。支付场景传 `from='payment'`，个人中心传 `from='mine'`。后续加直播打赏只需传新参数，充值代码零改动。

**追问：from 参数多了不会变成 if-else 地狱吗？**
当前 2 个分支可控，超过 3 个时可以抽象为策略注册表：`{ payment: () => pushReplacementNamed('paymentDetail'), mine: () => pop() }`，根据 from 查表执行。这是渐进式重构，不需要一步到位。

---

### H6. 优惠券为什么分 coupons 和 user_coupons 两张表？

**答辩精简回答**（30 秒版）
模板-实例分离：coupons 是券模板（面额、门槛、总量），user_coupons 是用户持有的实例（是否已使用）。合并成一张会导致总量统计需要更新 N 条记录、用户管理券实例时影响统计数据、加"每人限领 N 张"约束时无法建唯一索引。

---

### H7. 优惠券的 4 个 Riverpod Provider 为什么不能合并？

**答辩精简回答**（30 秒版）
合并会导致两个问题：① 性能——Map 结构的商品券选择状态任何变化都会触发满减券和商品详情券的消费者重建；② 逻辑耦合——三种券的清除时机不同（商品券清全选时清除、满减券结算时清除），放在一起容易误清。Riverpod 的核心理念是小 Provider 组合优于大 Provider 继承。

---

### H8. 当前是模拟支付，接入真实微信支付需要改什么？

**答辩精简回答**（30 秒版）
三处改动：① 替换随机成功率为真实 SDK 调用；② 增加支付幂等性——`payment_id` 唯一标识防重复扣款；③ 增加异步回调处理——微信支付是异步通知，需要改为"发起支付→pending→回调更新状态"。最关键的是②和③，忽略它们会导致重复扣款或丢单。

---

### H9. 为什么用 fetch 裸调 LLM 而不用 LangChain 完整 SDK？

**答辩精简回答**（30 秒版）
当前只需要"发消息→收回复"一个能力，fetch 裸调 30 行代码搞定。LangChain 完整 SDK 引入后 node_modules 膨胀 50MB，依赖链复杂。当需要 Chain（多轮对话自动管理）、Agent（工具调用）、多模型路由时再切回——按需引入，渐进增强。

---

### H10. SQLite 同步 API 真的线程安全吗？为什么不用异步 API？

**答辩精简回答**（30 秒版）
better-sqlite3 的同步 API 在 Node.js 单线程模型下，`SELECT → UPDATE → INSERT` 三个操作之间不会被其他请求打断，天然避免了异步竞态。当前 Demo 阶段并发量低，同步 API 够了。如果并发量上去了，先调 WAL 模式 + 增大 cache，还不够再加 Redis 缓存或换 PostgreSQL。

---

## 三、拓展视野（5 题）

> 答辩官跳出你的模块，看你的全局思考能力。

### E1. 你的购物支付链路，和内容流（视频 Feed）、直播模块之间有哪些交互？

**答辩精简回答**（30 秒版）
三个交互点：① 视频/直播中的商品卡点击"立即购买"→ 通过路由参数把商品信息传入订单确认页；② 视频/直播中"加入购物车"→ 调用购物车 API，共享 cart_items 表；③ 订单列表中的商品图片点击 → 通过 product_id 跳回商品详情页。桥梁是 product_id 和 go_router 路由参数。

---

### E2. 日均 10 万订单，SQLite 撑得住吗？怎么演进？

**答辩精简回答**（30 秒版）
10 万订单/天 ≈ 每秒 1.16 单，高峰期 ×5 ≈ 每秒 5-10 个写操作，加上购物车、退款、客服消息，峰值约 50 写/秒。SQLite WAL 模式能应付。演进路径：① 开 WAL + 增大 cache（零成本）→ ② 读写分离 → ③ 核心表迁 PostgreSQL + Redis。第一步和第二步之间至少差一个数量级。

---

### E3. 如果 LLM API 密钥过期或服务宕机，客服系统还能用吗？

**答辩精简回答**（30 秒版）
能用但降级为纯人工模式。askAI() 的 catch 块返回兜底文案 + 引导转人工，页面不崩溃。但当前降级边界是用户必须主动点转人工——改进方向是检测连续 3 次 AI 失败后自动触发转人工。

---

### E4. 当前购物支付链路最大的技术债是什么？两周怎么改？

**答辩精简回答**（30 秒版）
服务端缺少超时订单自动取消机制——用户关掉 App 后订单永远 pending，库存被锁。两周计划：第一周实现 node-cron 定时任务（每 30 秒扫 3 分钟前的 pending 订单，释放库存、标记 cancelled）；第二周补测试和监控。优先级高于支付幂等，因为直接阻塞其他用户下单。

---

### E5. 3 个前端 + 2 个后端并行开发，哪里最容易冲突？

**答辩精简回答**（30 秒版）
订单确认页——同时服务购物车结算和立即购买两条入口，接收参数最多（from / total / count / 优惠参数 / 商品快照参数），多人同时加参数或改 UI 必冲突。解决：两类入口拆成独立构造函数工厂 + 参数对象化（OrderConfirmParams 类）。

## 附录：答辩前 5 分钟速记卡

| 关键概念 | 一句话速记 |
|---------|-----------|
| **商品快照** | 下单时复制商品信息存订单，不改不删——就像拍照片，人老了照片不变 |
| **先扣后退** | 支付前先扣余额，失败再还回去——防止两笔订单同时花同一笔钱 |
| **RAG** | 先搜知识库再让 LLM 回答——给 AI 一本教科书，不让他瞎编 |
| **策略模式** | 定义接口、换实现不改调用——VectorStore 从 Dummy 换成 Chroma 只需一行 |
| **依赖倒置** | 充值模块不依赖调用方，只暴露 from 参数——谁来调用我就听谁的 |
| **pushReplacement** | 替换当前页面而非叠加——充值完成后不让人看到中间页 |
| **状态机守卫** | 每次状态切换都有前置检查——pending 才能付、paid 才能收货 |
| **锚定效应** | 大额档的高赠送让小额档显得"亏"——¥648 送 10%，¥30 就只送 3% |
