# 第三阶段：不稳定原因分析与修复

> 2026-06-01

## 1. 内容流切换不稳定

### C1-1: 快速滑动时双 player 竞态 [P0]
- **位置**: `video_player_widget.dart:_initPlayer()` + `didUpdateWidget`
- **根因**: `didUpdateWidget` → `_releasePlayer()` → `_initPlayer()`。`_initPlayer` 中 `await pool.acquire()` 期间如果再次 rebuild，第二个 `_releasePlayer()` 释放第一个 acquire 拿到的 controller，然后又 `_initPlayer()`。最终两个 acquire 同时进行。
- **修复**: 添加 `_initToken` 取消机制 — 每次 init 递增 token，回调时检查 token 是否匹配。

### C1-2: PageView.itemBuilder 每次滑动都创建新 Widget [P1]
- **位置**: `feed_page.dart:312` PageView.builder
- **根因**: PageView 默认不 keepAlive 除当前页外的 item。滑回上一个视频时创建新的 VideoPlayerWidget → 重新 acquire → 重新 initialize。如果 PlayerPool 中还有缓存的 controller，acquire 是即时的；如果已被 evict，则需重新网络加载。
- **修复**: Pool size 已增大到 4，基本够用。额外增加 `PageView` 的 `allowImplicitScrolling: true`。

## 2. 实时通信不稳定

### C2-1: WebSocket 重连 Timer 重复创建 [P0]
- **位置**: `websocket_service.dart:_scheduleReconnect()`
- **根因**: `onDisconnect` 和 `onConnectError` 各触发一次 `_scheduleReconnect`，无去重 → 多个 Timer 同时运行 → 重复连接
- **修复**: `_scheduleReconnect` 开头检查 `if (_reconnectTimer?.isActive == true) return;`

### C2-2: 在线人数更新频率无节流 [P2]
- **位置**: `live_provider.dart:_handleEvent()`
- **根因**: 每次 `online_count` 事件都 setState → 如果服务端高频推送，UI 高频重绘
- **修复**: 仅在数值变化超过 10% 或间隔 > 3s 时更新

---

## 3. 商品卡交互不稳定

### C3-1: 直播间商品卡初始不显示 [P1]
- **位置**: `live_room_page.dart` Builder block
- **根因**: 进入直播间时 `state.currentProduct` 为 null 且 `state.products` 可能尚未加载（异步 API）。`enterRoom` 完成后 `products` 才填充，此时浮层才出现。
- **状态**: ✅ 已有兜底 `state.currentProduct ?? state.products.first`

### C3-2: 直播间商品卡偶现空白 [P1]
- **位置**: `floating_product_card.dart:build()`
- **根因**: `CachedNetworkImage` 加载失败无 fallback → 卡片上半部空白
- **修复**: errorWidget 已有占位图标

---

## 4. 购物链路不稳定

### C4-1: 支付后订单列表不刷新 [P1]
- **位置**: `order_provider.dart:loadOrders()`
- **根因**: `if (state.isLoading) return;` 阻止了即时刷新。支付完成后 `loadOrders()` 被调用，但如果此时 isLoading=true（前一页请求未完成），刷新被跳过。
- **修复**: 增加 `force` 参数跳过 isLoading 检查。

### C4-2: 购物车数量偶现错误 [P1]
- **位置**: `cart_provider.dart:updateQuantity()`
- **根因**: 先乐观更新 UI (`state.copyWith(items: updated)`)，再调 API。若 API 失败回滚，但期间用户可能已看到错误数量并离开页面。
- **状态**: ✅ 已有乐观+回滚机制，但时间窗口仍存在
