**为什么这样分层——核心设计理念：**

| 层次 | 选型 | 设计理由 |
|------|------|---------|
| **Page → Provider 单向依赖** | Riverpod 的 `ref.read/ref.watch` | 页面不直接操作状态，只通过 Provider 接口触发动作。Provider 是"中间人"——页面"说我要下单"，Provider 负责调 API、处理错误、更新状态、通知 UI。**逻辑与 UI 分离** |
| **Provider 通过构造函数注入 API** | Provider 内部 `new OrderApi(client)` | Provider 不直接 import DioClient，而是通过构造函数传入——单元测试时可以替换为 MockApi。**依赖倒置** |
| **所有 API 共享 DioClient** | 单例模式注入 | 一个 DioClient 统一管理 Auth Token 注入、请求日志、重试策略、异常映射。路由层 `service_providers.dart` 注册一次，所有 Api 类复用。**避免重复配置** |
| **API 层不处理业务逻辑** | 仅做 HTTP + JSON 序列化 | Api 层是"邮递员"：把请求体发出去，把响应体转成 Model 返回。不关心"库存够不够"、"优惠券能不能用"。**职责单一** |
| **服务端不显式分层** | 路由直调 SQL | Node.js 端没有 Service 层（除 RAG 客服外），路由函数直接操作 `better-sqlite3` 同步 API。原因：SQLite 同步调用天然事务安全，加 Service 层反而增加跳转层级。**场景驱动，不追求过度设计** |

## 二、关键分层选型决策详解

### 2.1 为什么 Provider 分拆成多个而非一个？

购物支付链路涉及 4 个核心 Provider，分别管理独立状态域：

```
┌──────────────────────────────────────────────────────────────────┐
│  为什么拆成4个？—— 作用域 × 生命周期 × 变更影响面                    │

│                                                                    │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │ OrderNotifier (订单)             作用域：全局                │  │

│  └────────────┬────────────────────────────────────────────────┘  │
│               │                                                  │
│  ┌────────────▼────────────────────────────────────────────────┐  │
│  │ CartNotifier (购物车)         作用域：购物车页面级             │  │
│  │ 状态：购物车列表、选中状态、总价                              │  │
│  │ 设计亮点：乐观更新 + 失败回滚                                 │  │
│  └────────────┬────────────────────────────────────────────────┘  │
│               │                                                  │
│  ┌────────────▼────────────────────────────────────────────────┐  │

│  └────────────┬────────────────────────────────────────────────┘  │

### 2.2 为什么服务端选 `better-sqlite3`（同步 SQL）而非异步 ORM？

```
异步 ORM (Prisma / TypeORM)          better-sqlite3 (同步)
┌──────────────────────┐            ┌──────────────────────┐
│  await db.order.     │            │  db.prepare('UPDATE   │
│    create({...})     │            │    orders SET         │
│  await db.inventory  │            │    status = ?         │
│    .update({...})    │            │    WHERE id = ?       │
│  await db.coupon.    │            │  ').run(status, id)  │
│    .update({...})    │            │                       │
│                      │            │                      │
│  每个 await 都是      │            │  同步 API = 天然事务   │
│  一次事件循环让步      │            │  两行代码不会被打断   │
│  三行 await 之间      │            │  不需要事务装饰器     │
│  可能插入其他操作      │            │                       │
└──────────────────────┘            └──────────────────────┘

┌──────────────────────────────────────────────────────────────────┐
│  支付场景的并发风险：                                              │
│                                                                    │
│  两笔订单同时请求抖币支付：                                        │
│  用户余额 = 100，订单A = 60，订单B = 60                          │
│                                                                    │
│  异步模式：                                                        │
│  ① 订单A: SELECT balance → 100 ≥ 60 → 通过                       │
│  ② 订单B: SELECT balance → 100 ≥ 60 → 通过 (脏读！)               │
│  ③ 订单A: UPDATE balance = 40                                     │
│  ④ 订单B: UPDATE balance = 40 (应为 -20，但少了事务约束)          │
│                                                                    │
│  同步模式：                                                        │
│  ① 订单A: UPDATE balance = balance - 60 → 100-60=40 (原子操作)   │
│  ② 订单B: UPDATE balance = balance - 60 → 40-60=-20 (被阻塞到①完成)│
│  ③ 检查余额 → 发现不足 → 回滚                                     │
│                                                                    │
│  结论：同步 SQL 的 UPDATE 是天然的排它锁，无需额外加锁机制          │
└──────────────────────────────────────────────────────────────────┘
```

### 2.3 为什么 `OrderConfirmPage` 要从 `from` 参数区分数据源？

```
                       ┌────────────────────────────────────────┐
                       │  问题根源：「功能耦合」                   │

                     │                   实现方案                        │
                     ▼                                                  │
┌──────────────────────────────────────────────────────────────────────┐
│  身份识别：widget.from 参数                                           │
│                                                                      │
│  ┌─ from='cart' ──────────────────────────────────────────────────┐ │

**路由选型理由（go_router）：**

| 特性 | 购物支付链路中的用途 |
|------|-------------------|
| **`pathParameters`** | `orderId`、`productId` 等路径参数天然匹配 RESTful 风格 |
| **`queryParameters`** | `from` 来源标识、金额、优惠券 ID 等附带信息通过查询参数传递 |
| **`pushReplacementNamed`** | 支付成功/超时跳转时替换当前路由，防止"返回键回到支付页" |
| **`goNamed`** | 支付成功后跳回订单列表，一次清除中间所有路由栈 |

## 五、技术栈选型理由

### 5.1 移动端（Flutter）

| 技术 | 在购物支付中的角色 | 为什么选它 |
|------|-----------------|----------|
| **Riverpod** | 订单状态管理 (`OrderNotifier`) / 购物车状态 (`CartNotifier`) / 用户余额 (`UserNotifier`) | `ref.watch` 精准追踪——订单状态变化只刷新订单页，不触发购物车重建。`StateNotifier` 天然支持乐观更新。比 Bloc 少 50% 样板代码 |
| **go_router** | 支付链路页面跳转 (`orderConfirm` → `paymentDetail` → `paymentResult` → `order`) | `pushReplacementNamed` 在支付成功后替换路由，用户按返回键不会回到支付页。`queryParameters` 透传 `from` 参数 |
| **Dio** | 所有订单/购物车/充值的 HTTP 请求 | 拦截器实现统一 Token 注入、请求日志、重试。`RetryInterceptor` 在支付请求超时时自动重试，避免用户看到"网络错误"白屏 |
| **cached_network_image** | 商品封面/订单商品图片展示 | 订单列表频繁上下滚动，缓存商品图片避免重复下载。退款页展示商品时也依赖此组件 |

### 5.2 服务端（Node.js）

| 技术 | 在购物支付中的角色 | 为什么选它 |
|------|-----------------|----------|
| **Fastify** | 订单/充值/客服路由 | 性能是 Express 的 2-3 倍，JSON Schema 内置校验——`POST /orders` 自动校验请求体格式。插件化架构，路由文件 `orders.ts` 即插即用 |
| **better-sqlite3** | 订单/商品/用户余额持久化 | **同步 API**——支付事务不需要 `await`，`UPDATE coin_balance -= amount` 天然原子执行。零配置部署，不需要安装数据库服务。单机场景下性能优于 PostgreSQL |
| **crypto** | 订单 ID 生成 (`crypto.randomUUID()`) | Node.js 内置模块，零依赖。UUID 格式的订单 ID 便于调试和索引 |

## 六、关键设计模式

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| **依赖倒置** | Provider → API 构造函数注入 | `OrderNotifier` 不 import `OrderApi`，而是通过构造函数接收。单元测试时传入 `MockOrderApi`，不依赖 Dio 和网络 |
| **策略模式** | `IVectorStore` 接口 + `DummyVectorStore` 实现 | 智能客服的向量检索定义接口，当前全量返回，未来可切换 ChromaDB，`langchain_cs.ts` 零修改 |
| **观察者模式** | Riverpod `ref.watch` | `OrderPage` `watch(orderProvider)` 监听订单列表变化自动刷新 UI。支付成功后 Provider 更新状态，订单列表自动显示最新状态 |
| **乐观更新 + 失败回滚** | 购物车 Provider | 先改本地 state 让用户立刻看到反馈，API 失败后 state 回滚。购物车加减数量、选中/取消选中均采用此模式 |
| **先扣后退** | 抖币支付 | 先 `UPDATE` 扣减余额（原子操作），支付失败再退。消除了"查余额→扣余额"的时间窗口，这是消除并发竞态的关键设计 |

## 七、总结

本购物支付链路的代码结构选型遵循两个核心原则：

1. **分层职责单一**：Page 只管 UI 和跳转、Provider 管状态与业务编排、API 管 HTTP 通信、Route 管请求处理、DB 管持久化——每一层不越界。改 UI 不改业务逻辑，改后端不影响前端。

2. **场景驱动，不追求过度设计**：
   - 服务端没有 Service 层——因为 SQLite 同步 API 足够直接，加 Service 反而增加跳转层级
   - Provider 拆成 4 个而不是 1 个——因为"选券"和"提交订单"是不同作用域的状态，拆开让变更影响面最小
   - `from` 参数区分数据源——因为购物车结算和立即购买是两种完全不同的交易场景，**提前分离** 比业务复杂后再重构成本低得多

用一句话概括：**Page 说"我要下单"，Provider 说"我来安排"，API 说"我发请求"，服务端说"我落数据"。各司其职，互不越界。**
