## 角色
你是一名**极端严格的后端持久化专家**，项目使用 **Dart** 语言（Dart Frog 框架）。你的任务是：**将当前基于内存模拟数据的 RESTful API 改造成真正的持久化存储**，所有业务数据在服务重启后不能丢失。

## 核心要求
- 选择一种持久化方案：**SQLite**（推荐，无需额外安装数据库服务）或 **PostgreSQL/MySQL**（需提供连接配置）或 **MongoDB** 或 **JSON 文件**（最简单，但性能差，仅作为备选）。
- **必须保证以下数据可持久化**：商品、视频、购物车、订单、评论。用户数据也可选持久化。
- 服务启动时自动创建数据库表/集合结构；关闭或重启后数据依然存在。
- 现有所有 API 端点（用户、视频、商品、购物车、订单、评论）的行为必须完全保持不变，仅将数据源从内存 List 改为持久化数据库。
- 所有功能（分页、搜索、购物车增删改查、订单创建与状态变更等）必须正确读写持久化数据。

## 详细要求

### 1. 数据库选型建议（按推荐顺序）
- **首选：SQLite**（使用 `sqflite` 或 `drift` 或 `sqlite3` + `sqflite_common_ffi`，因为 Dart Frog 运行在 Dart VM 上，可以使用 `package:sqlite3` 和 `package:sqljocky` 等）。推荐 `package:sqflite_common_ffi`（跨平台）。
- **备选：PostgreSQL**（使用 `package:postgres`），需提供环境变量配置。
- **备选：JSON 文件**（使用 `dart:io` 读写文件，每次修改后覆盖保存，启动时加载）。这种方式最简，但并发写入需谨慎。

### 2. 数据模型对应表结构（或集合）
- **users**：id, username, password, nickname, avatar, createdAt
- **videos**：id, title, cover, playUrl, likeCount, commentCount, productIds (JSON 或关联表)
- **products**：id, title, price, discountPrice, image, sales, stock, tags (JSON), specs (JSON), isExplaining
- **cart_items**：id, userId, productId, quantity, selected, spec (JSON)
- **orders**：id, orderNo, userId, totalAmount, discountAmount, shippingFee, payAmount, status, address, createTime
- **order_items**：id, orderId, productId, productName, price, quantity, total (一对多关联)
- **comments**：id, targetType, targetId, userId, nickname, content, createTime

### 3. 实现细节
- 在 `lib/data/database.dart` 中替换内存存储为数据库操作。
- 提供初始化方法：建表/建索引，如果表为空则插入示例数据（至少 2 用户、3 商品、2 视频、若干购物车项和订单、评论）。
- 所有 CRUD 必须使用异步方法（Future）。
- 对于 SQLite，使用 `package:sqflite_common_ffi`（在 Dart VM 中工作），并在 `main` 中初始化数据库文件（例如 `app.db`）。
- 对于 JSON 文件方案：将每个表存储为一个独立 JSON 文件（如 `users.json`），启动时读取，每次修改后完整写入（注意并发锁，可以使用简单同步锁或 `mutex`）。

### 4. 改造范围
需要修改的文件：
- `lib/data/database.dart`：完全重写为持久化版本。
- `lib/routes/api/*.dart`：将所有直接操作 `Database` 静态列表的地方改为调用异步数据库方法（例如 `await db.insert(...)`、`await db.query(...)`）。
- `pubspec.yaml`：添加所选数据库驱动的依赖。

### 5. 验收标准
- 启动服务，执行任意修改数据的 API（如添加购物车、创建订单、发布评论）。
- 停止服务（Ctrl+C）。
- 重新启动服务，再次查询同一资源，之前修改的数据必须仍然存在。
- 所有原有 API 测试用例（curl 命令）必须仍能返回符合预期的结果。

## 硬性约束（违反即为不合格）
- ❌ 不得保留任何数据在内存变量中（例如全局 `List`）作为主要存储。数据库必须成为唯一真实来源。
- ❌ 不得因为引入数据库而破坏原有 API 的响应格式或状态码。
- ❌ 不得丢失任何业务逻辑（如购物车数量修改时的总价计算、订单支付状态流转）。
- ❌ 对于 SQLite，必须处理并发请求（同一数据库连接可用，sqflite 内部队列化）。
- ❌ 必须包含示例数据的初始化，便于演示。
