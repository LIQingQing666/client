**下单链路数据流转说明**：
1. **数据源鉴别**：`OrderConfirmPage._getDisplayItems()` 通过 `widget.from` 区分数据来源——`from='cart'` 从购物车 Provider 读取已选商品；`from='buy_now'` 从路由参数构建单商品快照（`CartItemModel` 兜底）。
2. **优惠券计算链**：`order_confirm_page.dart` 先读 `selectedCartItemCouponsProvider` 计算每件商品的 `itemDiscounts`（商品券），再结合 `payAmount`（含满减券）一并提交。计算顺序：商品总额 → 商品券 → 券后小计 → 满减券 → 实付金额。
3. **服务端零信任**：`orders.ts` 从数据库查 `products.price` 重算 `totalAmount`，前端传的 `pay_amount` 仅用于计算 `discountAmount = totalAmount - payAmount`，杜绝篡改。
4. **成功跳转**：`createOrder()` 返回后 `context.pushReplacementNamed('paymentDetail', {orderId, amount})`。

### 二、支付链路（支付详情 → 支付结果 → 订单列表）

```
┌───────────────────────────────────────┐
│   OrderPage (订单列表页)                │
│  apps/mobile/lib/pages/order/         │
│  order_page.dart                      │
│                                       │
│  功能：Tab筛选 + 订单操作               │
│    Tab: 全部/待支付/已支付/已完成         │
│    待支付 → 点击"去支付"                │
│    已支付 → 点击"确认收货"              │
│    已完成 → 点击"退货/退款"或"售后客服"  │
└────────┬──────────────────────────────┘
         │ pushNamed('paymentDetail', {orderId, amount})
         ▼
┌──────────────────────────────────────────────────────────────┐
│              PaymentDetailPage (支付详情页)                    │
│  apps/mobile/lib/pages/order/payment_detail_page.dart        │
│                                                               │
│  关键方法/流程：                                                │
│    initState()                                                │
│      ├── _startCountdown()  ← 180秒倒计时，最后30秒变红       │
│      └── _loadOrderDetail() ← 读订单详情展示商品信息           │
│                                                               │
│    _doPay()  ← 核心支付方法                                    │
│      ① 检查支付方式                                            │
│      ② if (coin支付) 检查 coinBalance ≥ amount               │
│      ③ 余额不足 → _showInsufficientBalanceDialog()            │
│      ④ 充足 → ref.read(orderProvider.notifier).payOrder()    │
│                                                               │
│    _showInsufficientBalanceDialog()                            │
│      → pushNamed('coinRecharge', {from:'payment', orderId})  │
└────────┬──────────────────────────────────┬───────────────────┘
         │ payOrder()                       │ 余额不足，跳转充值
         ▼                                  ▼
┌──────────────────────────┐    ┌──────────────────────────────────┐
│   OrderNotifier.payOrder │    │ CoinRechargePage (抖币充值页)     │
│   → OrderApi.payOrder()  │    │ apps/mobile/lib/pages/recharge/  │
│                          │    │ coin_recharge_page.dart          │
└────────┬─────────────────┘    │                                   │
         │ HTTP POST            │  充值成功后 → RechargeResultPage    │
         ▼                      │  根据 from='payment' 回跳支付页    │
┌───────────────────────────────┴──────────────────────────────────┐
│         routes/orders.ts (支付接口)                               │
│                                                                   │
│  POST /api/orders/:id/pay { payment_method }                     │
│    流程:                                                          │
│      ① 校验订单状态 (必须是 pending)                               │
│      ② if (payment_method === 'coin')                            │
│           → 先 UPDATE coin_balance -= payAmount (先扣)            │
│      ③ 模拟支付 (random() > 0.1 → 90%成功)                      │
│      ④ if (成功) → UPDATE orders SET status='paid'               │
│         else → status='payment_failed'                           │
│                → if (coin) UPDATE coin_balance += payAmount (后退)│
│      ⑤ 返回 {status, new_balance}                                 │
└────────┬─────────────────────────────────────────────────────────┘
         │ 支付结果返回客户端
         ▼
┌──────────────────────────────────────────────────────────────┐
│              PaymentResultPage (支付结果页)                    │
│  apps/mobile/lib/pages/order/payment_result_page.dart        │
│                                                               │
│  三态处理：                                                    │
│    paid    → ✓ 支付成功 + RecommendProducts 推荐               │
│    failed  → ✗ 显示"重新支付"按钮                               │
│    timeout → ⏰ 显示"返回购物车"按钮 (180秒倒计时触发)          │
│                                                               │
│  成功时：ref.read(orderProvider.notifier).requestSwitchTab(2) │
│  跳转：context.goNamed('order') ← 订单列表的"已支付"tab         │
└──────────────────────────────────────────────────────────────┘
```

**支付链路关键说明**：
1. **先扣后退原子性**：抖币支付时 `UPDATE coin_balance -= payAmount` 先执行（SQL 原子操作），支付失败再 `+= payAmount` 退还。消除了"查余额→扣余额"之间的并发窗口。
2. **充值闭环**：`from` 参数贯穿 `paymentDetail → coinRecharge → rechargeResult → paymentDetail` 全链路，解决"充完值回哪里"的问题。
3. **倒计时兜底**：`_countdownSeconds` 归零时自动跳转 `paymentResult?status=timeout`，防止订单永远卡在"支付中"。

│      → buildSystemPrompt(docsContext)                        ││
│      → callLLM(system + history + user) → reply              ││
│      → catch → "转人工" 兜底                                  ││
└──────────────────────────────────────────────────────────────┘│
                                                                 │
┌───────────────────────────────────────────────────────────────┐
│        db/schema.ts (数据库)                                   │
│                                                               │
│  refund_records 表:                                           │
│    id, order_id, user_id, product_id,                         │
│    product_name, quantity, refund_amount, reason, created_at  │
│                                                               │
│  与订单关联：refund_records.order_id → orders.id              │
│  每次退款只记录商品级别，不改变 orders.status                    │
└───────────────────────────────────────────────────────────────┘
```

**退款链路关键说明**：
1. **商品级退款**：不改变订单整体 `status`（保持 `completed`），仅通过 `refund_records` 表记录已退款商品，`refunded_product_ids` 通过 `SELECT DISTINCT product_id FROM refund_records WHERE order_id = ?` 关联查询。
2. **退款金额比例计算**：区分商品券折扣（独立全额退回）和满减券折扣（按券后价比例分摊），`refundAmount = Math.round((itemAfterProductCoupon - itemFullReductionShare) * 100) / 100`。
3. **AI 降级**：`langchain_cs.ts` 中 `callLLM()` 抛出异常时不抛给前端，直接返回"请稍后再试或转人工"兜底文案，前端 `isHumanService` 标记可切换到人工模式。
4. **转人工机制**：`transferToHuman()` 后 `customerServiceState.isHumanService = true`，前端不再发 AI 请求，只调 `reply()` 接收人工回复。
