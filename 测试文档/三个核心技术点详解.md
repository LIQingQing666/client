# 三个核心技术点详解

---

## 技术点一：PlayerPool 播放器池——四级策略 + 引用计数生命周期

### 要解决的问题

短视频上下滑场景中，每个视频需要一个 `VideoPlayerController`（底层对应一个 Android ExoPlayer 实例）。如果每次滑动都创建新 Controller、销毁旧 Controller：

- **性能差**：`VideoPlayerController.networkUrl().initialize()` 需要建立 HTTP 连接、下载首帧、初始化硬件解码器，耗时 2-10 秒
- **资源浪费**：ExoPlayer 是重量级系统资源，频繁创建/销毁导致 GC 压力和 GPU 内存抖动
- **并发超限**：MediaTek 芯片通常只支持 3-4 个并发硬件解码器，超出后新实例无法获取解码器

需要一个**复用机制**：已加载过的视频不要销毁，下次用到直接拿来用。

### 设计方案

**核心数据结构**：

```dart
poolSize = 3  // 全局最多同时存在 3 个 ExoPlayer
Map<String, _PooledPlayer> _players = {}
Set<String> _initializing = {}  // 防并发创建

class _PooledPlayer {
  VideoPlayerController controller;  // 真正的播放器实例
  String url;                        // 视频 URL（去重关键）
  int refCount;                      // 有多少个 VideoPlayerWidget 在用
}
```

**`acquire` 的四级回退策略**：

```
用户滑动到新视频
  │
  ▼
PlayerPool.acquire("video-abc", "https://...butterfly.mp4")
  │
  ├─ ① videoId 命中？
  │     _players["video-abc"] 存在？
  │     YES → refCount++ → 直接返回 controller（瞬时）
  │
  ├─ ② URL 去重命中？
  │     所有 10 个视频共用同一个 butterfly.mp4 URL
  │     遍历 _players（最多 3 个 entry）
  │     _players["video-xyz"].url == 当前 url？
  │     YES → refCount++ → _players["video-abc"] = 同一 _PooledPlayer 实例（别名）
  │     → 返回已有 controller（瞬时）
  │     → 不创建新 ExoPlayer！
  │
  ├─ ③ 正在初始化中？
  │     _initializing.contains("video-abc")？
  │     YES → 等待 200ms 后重试（防并发创建）
  │
  ├─ ④ 池满？
  │     _players.length >= poolSize (3)
  │     YES → 遍历找出 refCount 最小的 → dispose 其 controller → remove
  │
  └─ ⑤ 创建新实例
        _initializing.add("video-abc")
        VideoPlayerController.networkUrl(url)
        → await initialize().timeout(15s)
        → setLooping(true)
        _players["video-abc"] = _PooledPlayer(controller, refCount=1, url)
        _initializing.remove("video-abc")
```

**`release` 的引用计数管理**：

```dart
void release(String videoId, {bool dispose = false}) {
  final entry = _players[videoId];
  entry.refCount--;                        // 引用 -1

  if (entry.refCount <= 0) {
    entry.controller.pause();              // 暂停播放
    // ⚠️ 不 dispose！保留首帧和缓冲数据
    if (dispose) _disposePlayer(videoId);  // 只在明确要求时才销毁
  }
}
```

**关键决策：引用计数归零后 pause 不 dispose**

这是整个设计的核心。如果 refCount=0 就 dispose ExoPlayer，下次滑回来又要重新下载+初始化解码器。保留 controller（仅 pause）意味着：
- 已下载的视频数据保留在 ExoPlayer 的缓冲区中
- 硬件解码器虽然释放了（pause 也会释放部分资源），但重新 play 时恢复极快
- 下次 acquire 同一个 videoId 或相同 URL → 瞬时返回，不需要重新 `initialize()`

### 为什么 URL 去重是关键优化

所有 10 个种子视频使用同一个演示 URL（`butterfly.mp4`）。如果没有 URL 去重：

```
10 个视频 → 10 个 controller → 10 个 ExoPlayer → MediaTek 芯片崩溃
```

有了 URL 去重：

```
10 个视频 → 1 个 controller（因为 URL 相同）→ 10 个别名指向同一实例 → 1 个 ExoPlayer
```

`_players` Map 中有 10 个 key（10 个不同 videoId），但它们的 `_PooledPlayer` 引用都指向同一个对象，refCount 累加到 10。当用户在不同视频间切换，acquire/release 只改变 refCount 计数，**底层只有一个 ExoPlayer 在工作**。

### 并发初始化防护

快速上下滑时，用户滑过 video-A → video-B → video-A。第一次到 video-A 时调用 `acquire("A", url)`，初始化需要 2-10 秒。用户快速滑回 video-A 时，第一次的 `acquire` 还没完成，`_players` 中没有 "A" 的记录。如果再次调用 `acquire`，会创建**第二个** controller 用于同一视频。

`_initializing` 集合跟踪正在进行的初始化：

```dart
if (_initializing.contains(videoId)) {
  await Future.delayed(200ms);  // 等待前一个初始化完成
  if (_players.containsKey(videoId)) {
    // 前一个初始化已完成 → 直接复用
    _players[videoId]!.refCount++;
    return _players[videoId]!.controller;
  }
}
```

### 达到的效果

| 指标 | 无 PlayerPool | 有 PlayerPool |
|------|-------------|-------------|
| 滑回已看视频 | 重新下载，2-10s | 瞬时（缓存命中） |
| 最多 ExoPlayer 数 | 视频总数（10+） | 3（池大小上限） |
| 同一 URL 的实例数 | N 个 | 1 个（URL 去重） |
| 快速滑动稳定性 | 并发初始化 crash | 防并发保护 |

## 技术点二：PiP 小窗播放——跨页面 Controller 所有权转移

### 要解决的问题

直播间视频正在播放，用户点击购物车或按返回键 → 视频不应该停止，而是变为悬浮小窗继续播放。用户点击小窗 → 返回全屏直播间，视频应该**无缝续播**（不重新下载、不重新缓冲、不跳帧）。

这是一个**跨页面共享同一 ExoPlayer 实例**的问题。Flutter 中同一个 Widget 不能在两个页面中同时存在，所以 Controller 的**所有权**需要在页面之间转移。

### 设计方案

**4 个文件协同**：

```
main.dart                     ← 全局浮窗挂载点（Stack 最顶层）
  │
pip_provider.dart            ← 全局状态机（唯一真理源）
  │
live_room_page.dart          ← 直播间页面（Controller 的原主人）
  │
floating_video_player.dart   ← 浮窗 UI（Controller 的临时主人）
```

**PipState 状态机**：

```
                    ┌────────────────────────┐
                    │   isActive: false       │
                    │   controller: null      │
                    │   roomInfo: null        │
                    └──────┬─────────────────┘
                           │ enterPip(ctrl, room)
                           ▼
                    ┌────────────────────────┐
                    │   isActive: true        │
                    │   controller: ctrl ←─────── 直播间的 controller
                    │   roomInfo: room        │
                    └──────┬─────────────────┘
                           │
              ┌────────────┼────────────┐
              │ 用户点 ×     │ 用户点浮窗  │
              ▼             ▼            │
     closePip()       exitPip()          │
     dispose ctrl     isActive=false     │
     清空 state       controller 保留 ───┘
                             │
                             ▼
                    ┌────────────────────────┐
                    │   isActive: false       │
                    │   controller: ctrl ←────── 保留！等直播间来取
                    │   roomInfo: room        │
                    └──────┬─────────────────┘
                           │ 直播间 initState 读到这个状态
                           │ → 取走 controller
                           ▼
                    releaseController()
                    → 清空 state → 直播间重新成为主人
```

**直播间 → 小窗（4 个入口统一处理）**：

```dart
// 入口 1：系统返回键
PopScope(
  canPop: false,
  onPopInvokedWithResult: (didPop, _) {
    if (_videoController != null && _videoReady) {
      ref.read(pipProvider.notifier).enterPip(_videoController!, _currentRoom);
    }
    context.pop();
  },
)

// 入口 2：购物车按钮
onTap: () {
  if (_videoController != null && _videoReady) {
    ref.read(pipProvider.notifier).enterPip(_videoController!, _currentRoom);
  }
  AppRouter.router.go('/cart');
}

// 入口 3：商品立即购买
// 入口 4：左上角返回箭头
```

所有入口使用 `_currentRoom` getter（`liveState.room ?? widget.room`），API 未完成时也有 fallback。

**直播间 dispose 时保护 Controller**：

```dart
void dispose() {
  final pipActive = ref.read(pipProvider).isActive;
  if (!pipActive) {
    _videoController?.pause();
    _videoController?.dispose();  // 只有不进入 PiP 时才销毁
  }
  // 如果 pipActive=true → Controller 被 PiP 接管，不销毁
}
```

**小窗 → 直播间（Controller 零开销转移）**：

```dart
// ① main.dart — 浮窗点击
onTap: () {
  ref.read(pipProvider.notifier).exitPip();  // isActive=false，保留 ctrl
  AppRouter.router.pushNamed('liveRoom', {roomId});
}

// ② live_room_page.dart — 新页面 initState
void initState() {
  final pipState = ref.read(pipProvider);
  if (!pipState.isActive && pipState.videoController != null) {
    _videoController = pipState.videoController;  // 取走！
    _videoReady = true;
    _videoController!.play();                      // 继续播放
    ref.read(pipProvider.notifier).releaseController(); // 清空 PiP 状态
    return;  // 跳过 _initVideo() → 不重新下载视频！
  }
  _initVideo(widget.room.videoUrl);  // 正常初始化
}
```

### 调试过程中发现的 Bug

**Bug：小窗返回后黑屏**

现象：点击浮窗"返回直播间" → loading 转圈 → 视频不播放。

排查过程：

1. 首先确认浮窗中视频正常播放 → PiP 保存 controller 成功 ✓
2. 然后检查返回后新页面是否读取了 controller → 发现 `initState()` 中的复用逻辑根本没执行
3. 为什么没执行？因为 `build()` 中有两个早期 return：
   ```dart
   if (state.room == null && errorMessage != null) return 错误页;
   if (state.room == null) return loading 转圈;
   ```
   从 PiP 返回时 `enterRoom()` API 调用还没完成，`state.room` 为 null → 直接显示 loading/错误页 → **`_LiveRoomActiveContent` 根本没创建** → `initState()` 不执行 → PiP controller 永远没机会被复用。

这是一个**Widget 生命周期 + 异步状态 + 条件渲染**的三重交叉 Bug。修复涉及 `live_room_page.dart` 中 7 处修改，核心思想是**将视频播放与房间数据加载解耦**——有 PiP controller 就立即展示视频，不用等 API 返回。

## 技术点三：跨栈调试——从 Dart 类型系统到 Android 硬件解码器

### 为什么这是一个亮点

这个项目中的 bug 不是"逻辑写错了"，而是跨越多层抽象的**系统边界问题**。解决它们需要理解 Dart 语言特性、Flutter 渲染管线、Android 平台视图、MediaTek 芯片驱动——四个完全不同的技术栈。

### 案例 1：红屏 crash——Dart `Error` ≠ `Exception`

**现象**：收藏页点击视频 → 页面永久转圈，无任何错误提示。

**排查**：

1. 加 `debugPrint` 发现 `_load()` 方法在 API 调用后立刻终止——没执行到 controller 创建
2. 检查 API 响应：HTTP 200，但 body 是 `{"code": 404, "data": null}`
3. 检查 `video_api.dart` 第 33 行：
   ```dart
   response.data!['data'] as Map<String, dynamic>
   ```
   `data` 为 null → `as Map<String, dynamic>` → `TypeError: Null is not a subtype of Map`
4. 但这个 `TypeError` **没有被捕获**！为什么？
   ```dart
   try { ... await api.getVideoDetail(id); ... }
   on Exception { setState(() => _videoError = true); }
   ```
5. 关键发现：Dart 的类型体系中——
   ```
   Exception  ←  catchable by "on Exception"
   Error      ←  NOT catchable by "on Exception"!
     ├── TypeError
     ├── NoSuchMethodError
     └── AssertionError
   ```
   `on Exception` 只能捕获 `Exception` 及其子类，`Error` 完全不在此分支中。

**教训**：Dart 的 `Error` 和 `Exception` 是完全独立的继承树。`Error` 表示程序 bug（不应被捕获），`Exception` 表示可恢复的运行时错误。但在生产环境中，API 返回 null 数据是可能发生的，应该用 `catch(e)` 全捕获，同时在 API 层做 null 防护。

### 案例 2：无法滑动——Android `SurfaceView` 触摸层级

**现象**：直播间多房间切换的 `PageView` 无法接收垂直滑动手势。用户上下划 → 没有任何反应。

**排查**：

1. 先用 `Flutter Inspector` 检查 Widget 树 → `PageView` 正确存在，`scrollDirection: Axis.vertical` 正确设置
2. 怀疑 `PopScope(canPop: false)` 拦截手势 → 改为 `true` → 问题依旧
3. 用 Android Layout Inspector 查看视图层级 → 发现一个 `SurfaceView` 占据整个屏幕，**Z-order 在所有 Flutter Widget 之上**
4. `VideoPlayer` Widget 底层通过 `SurfaceView` 渲染视频。`SurfaceView` 是 Android 原生视图，拥有独立的 Window，**所有触摸事件先到达 SurfaceView，不经过 Flutter 的 GestureDetector**
5. Flutter 的 `PageView` 在 Flutter 渲染层（Skia），位于 SurfaceView **下方** → 永远收不到触摸事件

**解决方案**：

```dart
// ① 用 IgnorePointer 包裹 VideoPlayer → Flutter 事件的 hit test 跳过它
IgnorePointer(
  child: VideoPlayer(controller),
)

// ② 添加独立的 GestureDetector 用于暂停/播放
GestureDetector(
  behavior: HitTestBehavior.translucent,  // 关键！
  onTap: () => togglePlayPause(),
  child: SizedBox.expand(),
)
```

`HitTestBehavior.translucent` 的精妙之处：
- `opaque`：吸收所有事件 → 拖拽无法穿透到 PageView
- `translucent`：参与 hit test 但不阻挡 → tap 被捕获（暂停/播放），drag 穿透到 PageView（切换房间）
- `deferToChild`：不参与 hit test → 无法捕获任何事件

**教训**：Flutter 的 `VideoPlayer` 在不同平台上的实现不同。Android 使用 `SurfaceView`（独立 Window，拦截触摸），iOS 使用 `Texture`（与 Flutter 共享 Window，不拦截）。同一个 Widget，跨平台行为可能完全不同。

### 案例 3：视频卡顿——MediaTek 硬件解码器上限

**现象**：快速上下滑后视频加载变慢，日志出现 `inputFps=0, outputFps=0`，`BufferPool evictor expired`。

**排查**：

1. `inputFps=0` 说明 ExoPlayer 解码器已创建但不工作 → 不是网络问题
2. 检查设备芯片：`c2.mtk.avc.decoder` → MediaTek 硬件 H.264 解码器
3. 查阅 MediaTek 芯片文档 → 通常只支持 **3-4 个并发硬件解码器实例**
4. 当前架构：PlayerPool（3 个 controller）+ 预加载（1 个）= 最多 4 个 ExoPlayer
5. 但如果独立播放页（SingleVideoPlayerPage）也创建 controller → 5 个实例 → 超限 → 新实例无法获取解码器
6. `BufferPool evictor expired` → Android 图形缓冲区池溢出，强制驱逐旧缓冲区

**解决方案**：
- PlayerPool 池大小 4→3
- 预加载数 2→1
- 独立播放页 dispose 时**立即 pause+dispose controller**（不缓存）
- 重试按钮先 dispose 旧 controller 再创建新 controller

**教训**：移动端的 ExoPlayer 实例有严格的硬件上限（3-4 个 H.264 解码器）。Flutter 的播放器池设计必须考虑这个物理限制，不能仅基于软件层面的"引用计数"逻辑。

### 三层穿透总结

```
Dart 层：Error ≠ Exception → catch(e) 全捕获
  │
Flutter 层：SurfaceView 在 Skia 之上，拦截触摸 → IgnorePointer + translucent
  │
Android 层：MediaTek 解码器上限 3-4 个 → 控制 ExoPlayer 并发数
```

**每个问题都跨越了至少一层抽象边界，不能只在一个层级内解决。**
