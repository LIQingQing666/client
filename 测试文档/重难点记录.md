## 一、WebSocket 连接失败导致弹幕/评论完全不显示

### 问题背景
直播间需要实时弹幕（danmaku）和评论功能，依赖 Socket.IO 实现 WebSocket 双向通信。服务端使用 Fastify + Socket.IO，客户端使用 socket_io_client (Dart)。

### 问题现象
- 进入直播间后，屏幕没有任何弹幕滚动，评论区空白
- 客户端 WebSocket 状态始终在 reconnecting
- 服务端日志显示 Socket.IO 握手请求返回 HTTP 404
- 服务端定时发送的模拟弹幕（每5秒一条）全部收不到

### 根因分析
客户端 `app_constants.dart` 中 `wsUrl` 配置为：
```dart
static const String wsUrl = 'http://192.168.50.174:3000/api';  // ❌ 多了 /api
```
但服务端 Socket.IO 监听的是**根路径** `/socket.io`（默认配置，未指定 `path` 选项）。

客户端构造的实际请求 URL 为 `/api/socket.io/?EIO=4&transport=polling`，到达服务端后无任何路由匹配，Fastify 返回 404。

追溯 Git 历史发现：某次合并冲突解决时，`wsUrl` 被从正确的 `http://192.168.50.174:3000`（不带 `/api`）错误地同步成了与 `baseUrl` 相同的值。

> **关键知识点**：`baseUrl`（REST API）需要 `/api` 前缀是因为 Fastify 路由注册了 `/api/xxx` 路径。但 Socket.IO 是直接挂载在 HTTP Server 上的独立协议处理器，监听根路径 `/socket.io`，与 Fastify 的路由系统无关。因此 `wsUrl` 不能带 `/api`。

### 解决方案
1. 将 `wsUrl` 改回 `http://192.168.50.174:3000`（去掉 `/api` 后缀）
2. 在 `app_constants.dart` 中添加注释，防止后续合并时再次出错：
```dart
// NOTE: baseUrl includes /api prefix for REST routes (Fastify).
// wsUrl MUST NOT include /api — Socket.IO listens at root /socket.io.
static const String wsUrl = 'http://192.168.50.174:3000';
```

## 二、小窗播放（PiP）返回直播间视频无法继续播放

### 问题背景
直播间支持 PiP（Picture-in-Picture）功能：用户退出直播间时视频继续在悬浮小窗播放，点击小窗可返回全屏直播间继续观看。该功能涉及 4 个文件的协同：`pip_provider.dart`（状态）、`floating_video_player.dart`（UI）、`main.dart`（挂载点）、`live_room_page.dart`（进退逻辑）。

### 问题现象
- 退出直播间 → 小窗正常播放 ✅
- 点击浮窗"返回直播间" → 跳转到 LiveRoomPage → 显示 loading 转圈或错误页 → 视频不播放 ❌
- 即使点击"重新加载"，视频也加载不出来

### 根因分析
**三层根因**：

**第一层**：`_LiveRoomActiveContentState.build()` 中有两个早期 return：
```dart
if (state.room == null && errorMessage != null) return 错误页;
if (state.room == null) return loading 转圈;
```
当从 PiP 返回时，`enterRoom()` API 调用需要时间（或失败时无法完成），此时 `state.room` 为 null。这两个 early return 直接显示错误/loading 页，导致 `_LiveRoomActiveContent` 的 `initState()` 中**复用 PiP controller 的逻辑永远不会执行**。

**第二层**：所有进入 PiP 的入口（返回按钮、购物车按钮、PopScope 返回键、立即购买）都检查 `state.room != null` 才保存 PiP 状态。如果 API 未完成，room 为 null → PiP 不会创建 → 离开页面时 video controller 被 dispose。

**第三层**：`LiveRoomPage.build()` 中 `displayRooms` 的 fallback 链不包含 PiP room info：
```dart
final displayRooms = rooms.isNotEmpty
    ? rooms
    : (liveState.room != null ? [liveState.room!] : <LiveRoomInfo>[]);
```
当 `roomListProvider` 为空且 `liveState.room` 也为 null 时，`displayRooms` 为空 → 直接显示 loading 转圈。

### 解决方案
涉及 `live_room_page.dart` 7 处修改，核心思路是**视频播放与房间数据加载解耦**：

1. **引入 `effectiveRoom`**：`state.room ?? widget.room`，不再因 room 为 null 提前 return
2. **`hasVideo` 条件保护**：仅当没有 PiP 视频时才显示错误/loading 页
3. **新增 `_currentRoom` getter**：所有 PiP 入口统一使用，确保 API 未完成时也有 fallback
4. **`initState()` 检测 PiP**：有 PiP controller 时立即设 `_initialized = true`，跳过等待
5. **`displayRooms` 新增第三级 fallback**：`pipState.roomInfo`
6. **4 个 PiP 入口**全部去掉 `room != null` 检查，改用 `_currentRoom`
7. **`_LiveRoomActiveContent.dispose()`**：检查 `pipState.isActive`，PiP 活跃时不 dispose controller

## 三、收藏页点击视频/商品"跳转讲解"无法播放

### 问题背景
收藏页（FavoritesPage）展示用户收藏的视频和商品。点击视频应直接播放，点击商品应弹出详情弹窗并可"跳转到对应的视频讲解时间点"。原设计将收藏页的视频播放跳转到 FeedPage（`/play/:videoId` 路由），但 FeedPage 是为内容流浏览设计的，存在多个与单视频播放不兼容的问题。

### 问题现象
- 收藏页点击视频 tile → 跳转到 FeedPage → 视频不播放，显示列表第一个视频
- 收藏页点击商品 → 商品详情弹窗 → 点击"跳转到讲解 (00:XX)" → 跳转后视频不播放
- 多次修改后依然失效

### 根因分析
**多轮迭代暴露出的层层问题**：

**第一轮** — FeedPage `loadVideos()` 覆盖问题：
- FeedPage 的 `FeedNotifier` 构造时调用 `loadVideos()`，从 API 推荐列表加载视频
- 收藏的视频可能不在推荐列表中 → `pendingJumpVideoId` 找不到匹配 → 静默失败
- 尝试用 `insertVideoAtFront()` 插入目标视频，但 `loadVideos()` 完成后用新列表**整体覆盖**，刚插入的视频被清除（竞态条件）

**第二轮** — 逻辑死代码：
```dart
if (targetIndex >= 0) { jump... }
_pendingJumpVideoId = null;  // ← 在 if 外面！无论找到与否都清空
if (_pendingJumpVideoId == null) return;  // ← fallback 永远走不到
```

**第三轮** — 深层架构冲突：
- FeedPage 设计目标是"在连续列表中上下滑浏览"，不适合"播放单个指定视频"
- FeedPage 的 `PlayerPool`（4 个 controller 缓存）和预加载队列（2 个预加载）都是为列表浏览设计的
- 收藏页直接走 FeedPage 不仅状态管理混乱，还会干扰列表浏览的流畅性

**第四轮** — `TypeError` 被吞掉：
- 种子数据重建后视频 UUID 变化，旧收藏中的视频 ID 失效
- API 返回 `{"code":404, "data":null}` → `response.data!['data'] as Map<String, dynamic>` null 强转崩溃
- `single_video_player_page.dart` 使用 `on Exception` 捕获，但 `TypeError` extends `Error`（非 `Exception`），无法被捕获
- `_loading = true` 永远不变 → 永久转圈

### 解决方案
**最终方案：创建完全独立的 `SingleVideoPlayerPage`**

```
设计原则：共享 URL 级缓存（数据层），完全分离 State（控制层）

收藏页 → SingleVideoPlayerPage           Feed 页
  │         │                               │
  │         ├─ _urlCache (URL→controller)   ├─ PlayerPool
  │         ├─ 自己的 controller 生命周期    ├─ FeedState（列表/索引）
  │         └─ 不碰 FeedState/预加载         └─ 预加载队列
```

**核心实现**：

1. **URL 级全局缓存 `_urlCache`**（`Map<String, _CachedEntry>`）：
   - 同一视频 URL 全局只有一个 ExoPlayer 实例
   - 引用计数管理：打开 +1，关闭 -1，归零时 dispose
   - 与 Feed PlayerPool 完全独立，互不干扰

2. **独立 controller 生命周期**：
   - 不走 `PlayerPool`，自己管理 `VideoPlayerController`
   - dispose 时归还缓存或清理

3. **新增 `/video/:videoId` 路由**（`singleVideo`），收藏页导航到该路由

4. **三层异常保护**：
   - API 层：`video_api.dart` — `data` 为 null 时抛 `ClientException`
   - 加载层：`single_video_player_page.dart` — `catch(e)` 捕获所有类型（Error + Exception）
   - 超时层：API 10 秒超时 + controller 初始化 20 秒超时

5. **Seek 边界保护**：`clamp(0, 视频时长 - 0.5s)`，防止跳转到视频末尾之外

## 四、商品讲解时间超出视频时长

### 问题背景
商品关联的视频可以"跳转到讲解"——点击后视频自动 seek 到商品讲解片段的时间点。种子数据中定义了每个商品的 `highlight_time`（单位：秒）。

### 问题现象
- 演示视频（butterfly.mp4）只有约 7 秒
- 种子数据中 `highlight_time` 设置为 5-35 秒（如 15s, 22s, 28s, 35s 等）
- 点击"跳转到讲解"后视频 seek 到超出时长的位置，无画面或跳转失败

### 根因分析
1. 种子数据编写时未参考实际演示视频的时长，`highlight_time` 使用了演示视频不支持的较大值
2. 客户端 seek 逻辑未做边界保护，允许 seek 到视频时长之外的位置

### 解决方案
1. 种子数据修正：将所有关联视频的前 10 个商品 `highlight_time` 从 5-35 秒改为 1-6 秒范围：
   ```
   TWS降噪耳机: 15s → 3s    法式连衣裙: 22s → 5s   扫地机器人: 28s → 2s
   洁面泡沫:    8s  → 4s    户外帐篷:   35s → 6s   工学椅:     18s → 1s
   鸡胸肉:     12s → 3s    磁吸充电宝: 25s → 5s   宠物驱虫:    5s → 2s
   清洁套装:   30s → 4s
   ```
2. 客户端 seek 增加 clamp 保护（`video_player_widget.dart:84-87`）：
   ```dart
   final clamped = seekTo.clamp(0, (durSec - 0.5).ceil().clamp(0, durSec));
   ```

## 五、收藏页视频/Feed 页视频加载卡顿与资源竞争

### 问题背景
收藏页点击视频会创建独立的 `VideoPlayerController` 用于单视频播放。同时 Feed 页使用 `PlayerPool` 管理 4 个 Controller + 2 个预加载。两者可能同时持有多个 ExoPlayer 实例。

### 问题现象
- 收藏页点击视频后长时间加载不出来
- Feed 页面视频播放出现卡顿
- Android 系统日志频繁出现 `BufferPool evictor expired: N, evicted: N`
- 日志出现 `c2.mtk.avc.decoder inputFps=0, outputFps=0` 表示视频解码器无输出

### 根因分析
1. **ExoPlayer 实例竞争**：每个 `VideoPlayerController.networkUrl()` 创建一个底层 ExoPlayer 实例。Android 系统对 ExoPlayer 数量有限制（通常 3-6 个）。Feed PlayerPool（4 个）+ 预加载（最多 1 个）+ 收藏页（1 个）= 最多 6 个 → 超出系统限制
2. **带宽竞争**：多个 ExoPlayer 同时从网络下载视频数据（butterfly.mp4 ~2MB），抢占同一个 WiFi 通道
3. **GPU Buffer 竞争**：每个 ExoPlayer 申请 GPU 缓冲区用于视频解码，BufferPool 满了就驱逐旧缓冲区 → 视频解码中断 → 卡顿
4. **旧版代码每次新建 controller**：不走缓存，每次都重新下载视频

### 解决方案
1. **URL 级缓存**（`_urlCache`）：同一视频 URL 只创建一个 ExoPlayer，多个页面共享同一实例
2. **引用计数管理**：打开 +1，关闭 -1，归零时 dispose，避免残留
3. **从 PlayerPool 剥离**：独立播放页使用独立缓存，不与 Feed PlayerPool 争用控制器
4. **超时保护**：API 10 秒 + 初始化 20 秒，避免永久挂起

## 六、Dart Error/Exception 类型差异导致异常未被捕获

### 问题背景
Flutter/Dart 中 `Error` 和 `Exception` 是不同的继承链：
- `Exception` — 程序可恢复的运行时异常（如网络超时、IO 错误）
- `Error` — 编程错误（如类型转换失败 `TypeError`、空指针 `NoSuchMethodError`）

### 问题现象
`single_video_player_page.dart` 中 `_load()` 方法使用 `on Exception` 捕获异常：
```dart
try { ... await api.getVideoDetail(id); ... }
on Exception { setState(() => _videoError = true); }
```
当 API 返回 `data: null` 时，`video_api.dart` 中 `data!['data'] as Map<String, dynamic>` 抛出 `TypeError`。但 `TypeError` extends `Error`，不被 `on Exception` 捕获 → 未处理异常 → 页面永久 loading。

### 根因分析
```dart
// Dart 类层次：
class TypeError extends Error {}        // ← 不是 Exception!
class ApiException implements Exception {} // ← 是 Exception

// 错误的捕获方式：
try { ... } on Exception { ... }  // 只捕获 Exception，不捕获 Error

// 正确的捕获方式：
try { ... } catch (e) { ... }     // 捕获所有（Error + Exception）
```

### 解决方案
1. `video_api.dart`：`getVideoDetail()` 中 `data` 为 null 时主动抛出 `ClientException`（extends `Exception`），不再让 `TypeError` 向上传播
2. `single_video_player_page.dart`：`on Exception` → `catch (e)`，捕获所有异常类型

## 七、收藏页独立播放页视频解码器停滞（inputFps=0）

### 问题背景
在完成收藏页独立播放页（`SingleVideoPlayerPage`）开发后，视频播放在特定设备（Oppo + MediaTek 芯片）上仍然失败。该页面经过多轮架构调整：最初走 FeedPage 路由 → 独立 `VideoPlayerController.networkUrl()` → URL 缓存 `_urlCache` → 共享 Feed `PlayerPool` → 最终回到完全独立的 controller。

### 问题现象
- API 获取视频元数据成功（HTTP 200）
- Android 日志显示 ExoPlayer 解码器已创建：`c2.mtk.avc.decoder [1280*720]`
- 但解码器状态为 `inputFps=0, outputFps=0 renderFps=0, discardFps=0`（无任何帧流动）
- Oppo 系统监控上报 `VideoStutter`：`FpsArray#3/0/0/0/0/0`（仅 3 帧后完全停滞）
- `BufferPool evictor expired: N, evicted: N` 缓冲区驱逐频繁
- 用户界面：要么永久 loading 转圈，要么显示封面图但视频永远不开始播放

### 根因分析

**核心原因**：收藏页独立播放与 Feed 页播放用的是**同一视频 URL**（butterfly.mp4），但创建的是**不同的 ExoPlayer 实例**。

```
Feed 页 IndexedStack（始终存活）
  └── VideoPlayer → ExoPlayer #1 → 占用硬件解码器槽位

收藏页 → 独立播放页（push 到路由栈顶）
  └── VideoPlayer → ExoPlayer #2 → 抢不到解码器资源
```

**三层系统级资源竞争**：

1. **硬件解码器槽位耗尽**：
   - MediaTek `c2.mtk.avc.decoder` 通常只支持 2-4 个并发 H.264 解码实例
   - Feed 页 `PlayerPool`（4 个缓存 controller）+ 独立页（1 个）≥ 5 个 → 超限
   - 新 ExoPlayer 申请解码器 → 系统拒绝或降级 → `inputFps=0`（无帧输入解码器）

2. **GPU BufferPool 争抢**：
   - 每个解码器实例需要 GPU 缓冲区存储解码帧
   - Android `BufferPoolAccessor2.0` 管理的缓冲区有总量上限
   - Feed 页非活跃 controller 仍持有缓冲区（仅 pause 未 dispose）
   - 新解码器无可用缓冲区 → `evictor expired` → 旧缓冲区被驱逐 → 两边都卡

3. **IndexedStack 中未释放的旧 Controller**：
   - Feed 页在 `IndexedStack` 中保持存活，其 `VideoPlayerWidget` 的 controller 仅 `pause()` 未 `dispose()`
   - 即使 feed 页不可见，其 ExoPlayer 仍占用硬件解码器槽位和 GPU 缓冲区
   - 新推出的独立播放页需要创建新 ExoPlayer，但资源已被占用

**与正常工作的 Feed 页的差异**：
- Feed 页使用 `PlayerPool` 管理 controller 生命周期，同一视频 URL 只创建一个实例
- Feed 页内上下滑切换视频时，同一个 ExoPlayer 实例被复用（URL 去重）
- 独立页创建了新实例，绕过了 URL 去重逻辑

### 解决方案

**最终版设计原则**：完全独立 controller，dispose 时立即释放所有系统资源。

```dart
// 1. 不复用 PlayerPool，自建 controller
final controller = VideoPlayerController.networkUrl(Uri.parse(url));
controller.addListener(_onChanged);
await controller.initialize().timeout(Duration(seconds: 20));

// 2. dispose 时立即 pause + dispose，释放硬件解码器
@override
void dispose() {
  _controller?.removeListener(_onChanged);
  _controller?.pause();
  _controller?.dispose();  // 立即释放 ExoPlayer 和解码器槽位
  _controller = null;
  super.dispose();
}

// 3. 重试时先清理旧 controller
onPressed: () {
  _controller?.dispose();  // 先释放旧的
  _controller = null;
  setState(() { _loading = true; _videoError = false; _videoReady = false; });
  _load();  // 再创建新的
}

// 4. 监听 controller 错误状态（hasError）
void _onChanged() {
  if (v.hasError) {
    setState(() { _loading = false; _videoError = true; });
  }
}
```

**为什么这一版更稳定**：
- `dispose()` 中主动 `dispose()` controller → 立即释放 ExoPlayer 和硬件解码器槽位
- 不再走 `PlayerPool` → 不持有引用计数 → 不会残留未释放的 controller
- `catch(e)` 代替 `on Exception` → `TypeError` 等 Error 类型也能被捕获
- 重试前先 dispose 旧 controller → 避免并发 ExoPlayer 实例堆积

**已知局限**：
- 首次播放需从 GitHub 下载视频（~2MB），国内网络可能需 5-15 秒
- Oppo + MediaTek 设备上，如果 Feed 页同时有 3+ 活跃 controller，独立页可能仍无法获取解码器
- 彻底解决需要 Feed 页在路由变化时主动释放非活跃 controller（未来优化方向）
