# 电商直播短视频应用 — 视频流与直播间模块技术方案

> 答辩人负责模块：内容流浏览（Feed 短视频滑动）、直播间实时互动、视频预加载、播放器池、小窗播放（PiP）、商品讲解片段定位、收藏页独立播放

## 1. 项目背景与职责

本项目是一个电商直播短视频应用，用户可在推荐流中上下滑动观看短视频，也可进入直播间与主播互动、购买商品。

**本人负责的核心模块**：

| 模块 | 说明 |
|------|------|
| 内容流浏览 | FeedPage 上下滑切换、VideoPlayerWidget 播放器组件、FeedNotifier 状态管理 |
| 直播间实时互动 | LiveRoomPage 直播间页面、DanmakuOverlay 弹幕动画、WebSocket 实时通信、评论列表、在线人数/热度 |
| 播放器池 | PlayerPool 引用计数复用、URL 去重、LRU 淘汰 |
| 视频预加载 | VideoPreloadManager 优先级队列、WiFi 感知、串行加载 |
| 小窗播放（PiP） | PipProvider 全局状态、FloatingVideoPlayer 浮窗 UI、进出 PiP 流程、controller 生命周期保护 |
| 商品讲解片段定位 | seek 跳转、高亮闪烁反馈、边界 clamp 保护 |
| 收藏页独立播放 | SingleVideoPlayerPage 独立视频播放页、与 Feed 页完全解耦 |

## 6. 核心模块详细设计

### 6.1 FeedPage 滑动切换 (`feed_page.dart`)

- 使用 `PageView.builder` + `scrollDirection: Axis.vertical` 实现上下滑
- `onPageChanged` 触发 `FeedNotifier.setCurrentIndex(index)`
- 每个 `VideoPlayerWidget` 接收 `isActive` 参数，仅当前页且底部 Tab 聚焦时为 true
- 列表末尾插入 loading 指示器，自动触发 `loadMore()`
- 支持 `initialVideoId` 参数实现外部跳转到指定视频

### 6.2 FeedNotifier 状态管理 (`feed_provider.dart`)

```dart
final class FeedState {
  final List<VideoModel> videos;
  final int currentIndex;
  final bool hasMore;
  final int page;
  final FeedTab tab;        // recommend / follow
  final String? errorMessage;
}
```

核心方法：
- `loadVideos()`：根据 tab 调用 `getRecommend()` 或 `getFollow()`，加载完成后触发预加载
- `setCurrentIndex(index)`：更新当前索引 → `_preloadAround(index)` → 预加载后续 2 个视频 → `_cancelStalePreloads(index)` 取消远离的视频 → 距末尾 3 个时自动 `loadMore()`
- `toggleLike(videoId)`：调用 API 后乐观更新本地 like 状态
- `insertVideoAtFront(video)`：将单个视频插入列表首位（用于外部跳转）
- `switchTab(tab)`：切换推荐/关注 Tab，清空列表重新加载

### 6.3 播放器池 PlayerPool (`player_pool.dart`)

**设计目标**：解决 PageView 上下滑时视频播放器的创建/销毁性能问题。

```
池大小: 4 (poolSize)
数据结构: Map<String, _PooledPlayer>

_PooledPlayer {
  VideoPlayerController controller;  // 真正持有视频播放器
  String url;                        // 视频 URL（用于去重）
  int refCount;                      // 引用计数
}
```

**核心流程**：

```dart
Future<VideoPlayerController> acquire(String videoId, String url) async {
  // 1. videoId 命中 → refCount++ → 直接返回
  if (_players.containsKey(videoId)) { refCount++; return controller; }

  // 2. URL 去重：同一 URL 的不同 videoId 共享同一个 controller
  for (entry in _players) {
    if (entry.value.url == url) {
      entry.value.refCount++;
      _players[videoId] = entry.value;  // 别名指向
      return entry.value.controller;
    }
  }

  // 3. 池满 → 淘汰 refCount 最小的
  _evictIfNeeded();

  // 4. 创建新 controller
  final controller = VideoPlayerController.networkUrl(Uri.parse(url));
  await controller.initialize();
  await controller.setLooping(true);
  _players[videoId] = _PooledPlayer(controller, refCount: 1, url);
  return controller;
}

void release(String videoId, {bool dispose = false}) {
  entry.refCount--;
  if (entry.refCount <= 0) {
    entry.controller.pause();       // 暂停但不销毁
    if (dispose) _disposePlayer();  // 仅在明确要求时销毁
  }
}
```

**设计要点**：
- **引用计数归零后 pause 但不 dispose**：保留已解码的首帧和缓冲数据，下次 acquire 可直接复用
- **URL 去重**：所有视频使用同一个演示 URL（butterfly.mp4），避免创建重复的 ExoPlayer 实例
- **LRU 变体淘汰**：淘汰 refCount 最小的条目

### 6.4 视频预加载 VideoPreloadManager (`video_preload_manager.dart`)

**设计目标**：在当前视频播放时，提前加载后续视频的播放器，减少滑动切换时的等待时间。

```
FeedNotifier.setCurrentIndex(index)
  → _preloadAround(index)              // 预加载 index+1, index+2
  → _cancelStalePreloads(index)        // 取消 index 之前和 index+2 之后的
  → VideoPreloadManager.enqueue(videoId, url, priority)
      → 优先级队列 (_queue.sort by priority desc)
      → _processQueue()
          → 串行执行 (maxConcurrent = 1)
          → _preloadOne(task)
              → PlayerPool.preload(videoId, url)
                  → acquire → 下载首帧 → release
```

**核心特性**：

| 特性 | 配置 | 说明 |
|------|------|------|
| 预加载数量 | `preloadVideoCount = 2` | 预加载当前视频之后的 2 个 |
| 并发控制 | `maxConcurrent = 1` | 一次只下载一个视频，避免抢带宽 |
| WiFi 感知 | `wifiOnly = true` | 蜂窝网络自动暂停，切回 WiFi 自动恢复 |
| 启动延迟 | `startupDelay = 800ms` | 首视频加载后延迟 800ms 开始预加载，让当前视频优先完成初始缓冲 |
| 超时保护 | `preloadTimeout = 30s` | 超时后标记失败并取消 |

### 6.8 小窗播放 PiP

**涉及 4 个文件协同**：

| 文件 | 职责 |
|------|------|
| `pip_provider.dart` | 全局状态：isActive, videoController, roomInfo |
| `floating_video_player.dart` | 浮窗 UI：可拖拽、边界约束、16:9 比例 |
| `main.dart` | 全局挂载点：`MaterialApp.router.builder` 的 Stack 最顶层 |
| `live_room_page.dart` | 进入/退出 PiP 的 4 个触发点 |

**进入 PiP 的 4 个入口**（均在 `live_room_page.dart`）：
1. 系统返回键 → `PopScope.onPopInvokedWithResult`
2. 购物车按钮 → `enterPip` + `AppRouter.router.go('/cart')`
3. 商品"立即购买" → `enterPip` + 跳转订单确认页
4. 左上角返回箭头 → `enterPip` + `context.pop()`

**返回直播间流程**：
```dart
// main.dart — 浮窗点击回调
onTap: () {
  ref.read(pipProvider.notifier).exitPip();  // isActive=false, 保留 controller
  AppRouter.router.pushNamed('liveRoom', {roomId});
}

// live_room_page.dart — initState 复用 controller
if (!pipState.isActive && pipState.videoController != null) {
  _videoController = pipState.videoController;  // 直接复用
  _videoReady = true;
  _videoController!.play();
  ref.read(pipProvider.notifier).releaseController(); // 转移所有权
  return;  // 跳过 _initVideo，不重新加载
}
```

**Controller 生命周期保护**：
- `_LiveRoomActiveContent.dispose()` 中检查 `pipState.isActive`，PiP 活跃时**不 dispose** controller
- `releaseController()` 在 controller 被直播页接管后清空 PiP 状态
- `closePip()` 仅在用户点击浮窗关闭按钮时才暂停+销毁

### 6.9 收藏页独立播放 SingleVideoPlayerPage (`single_video_player_page.dart`)

**设计动机**：收藏页点击视频或商品的"跳转到讲解"需要一个轻量视频播放器。最初尝试复用 FeedPage（通过 `/play/:videoId` 路由），但 FeedPage 的 `loadVideos()` 流程会覆盖掉目标视频，且 FeedPage 的列表浏览逻辑与单视频播放不兼容。

**最终方案**：完全独立的单视频播放页，与 Feed 页零耦合。

```
架构：
  Feed 页 (IndexedStack)                收藏页 → SingleVideoPlayerPage
  ├── PlayerPool (4 槽位)                 ├── 自建 VideoPlayerController
  ├── VideoPreloadManager (2 预加载)       ├── 独立生命周期 (init → play → dispose)
  ├── FeedNotifier (列表/索引/分页)       ├── 不碰 FeedState
  └── FeedPage UI                        └── 自有 UI (封面、播放、seek、操作栏)
```

**关键实现**：
```dart
// controller 创建
final controller = VideoPlayerController.networkUrl(Uri.parse(url));
await controller.initialize().timeout(Duration(seconds: 20));

// dispose 立即释放硬件解码器
@override
void dispose() {
  _controller?.removeListener(_onChanged);
  _controller?.pause();
  _controller?.dispose();
  _controller = null;
  super.dispose();
}
```

## 7. 重难点问题与解决方案

### 7.1 WebSocket URL 错误导致弹幕完全不显示

- **问题背景**：客户端进入直播间后需要显示实时弹幕，依赖 Socket.IO WebSocket 连接。
- **问题现象**：评论区空白，无弹幕滚动，WebSocket 状态始终 `reconnecting`，服务端返回 404。
- **根因分析**：`app_constants.dart` 中 `wsUrl` 在合并冲突时被错误同步为 `http://192.168.50.174:3000/api`。服务端 Socket.IO 监听根路径 `/socket.io`，客户端请求路径变成 `/api/socket.io/...`，路由不匹配返回 404。
- **解决方案**：`wsUrl` 恢复为 `http://192.168.50.174:3000`（去掉 `/api` 后缀），添加注释防止再次出错。**关键知识点**：REST API 需要 `/api` 前缀，但 Socket.IO 直接挂载在 HTTP Server 上，与 Fastify 路由系统无关。

### 7.2 小窗返回直播间后视频无法播放

- **问题背景**：直播间的 PiP 功能：退出直播间时视频在悬浮小窗播放，点击浮窗应返回直播间继续播放。
- **问题现象**：点击浮窗"返回直播间"后，页面显示 loading 转圈或错误页，视频不播放。
- **根因分析**（三层）：
  1. `_LiveRoomActiveContent.build()` 中 `state.room == null` 提前 return 错误/loading 页，导致 `initState()` 中 PiP controller 复用逻辑永远不会执行。
  2. 所有 PiP 入口检查 `room != null` 才保存 PiP，但 `enterRoom` API 未完成时 room 为 null → PiP 未创建 → 离开页面时 controller 被 dispose。
  3. `LiveRoomPage.displayRooms` fallback 链不包含 `pipState.roomInfo`。
- **解决方案**（7 处代码修改）：
  - 引入 `effectiveRoom = state.room ?? widget.room`，不再因 room 为 null 提前 return。
  - 仅 `!hasVideo` 时显示错误/loading。
  - 新增 `_currentRoom` getter 统一 PiP 入口 fallback。
  - `initState` 检测 PiP controller 立即跳过加载等待。
  - `displayRooms` 新增 `pipState.roomInfo` 第三级 fallback。

### 7.3 收藏页视频无法播放（多轮迭代）

- **问题背景**：收藏页展示用户收藏的视频和商品，点击应直接播放视频或跳转到商品讲解片段。
- **问题现象**：点击视频/商品"跳转到讲解"后，FeedPage 不播放目标视频、视频加载缓慢、或永久显示 loading。
- **根因分析**（四轮发现）：
  1. FeedPage 的 `loadVideos()` 会用新视频列表整体覆盖 `insertVideoAtFront()` 刚插入的视频（竞态条件）。
  2. `_pendingJumpVideoId = null` 放在 `if (targetIndex >= 0)` 外面 —— 逻辑死代码，fallback 永远不执行。
  3. FeedPage 的列表浏览架构与单视频播放需求不兼容，PlayerPool 和预加载干扰独立播放。
  4. `on Exception` 不捕获 `TypeError`（Dart 中 `Error` ≠ `Exception`），API 返回 null 时异常未被捕获，页面永久 loading。
- **解决方案**：创建完全独立的 `SingleVideoPlayerPage`，自建 controller、独立生命周期、`catch(e)` 全捕获、API 超时保护。

### 7.4 ExoPlayer 硬件解码器资源竞争

- **问题背景**：Feed 页（IndexedStack 中保持存活）持有 4 个 ExoPlayer 实例，独立播放页创建新实例时可能超出硬件解码器限制。
- **问题现象**：视频解码器已创建（`c2.mtk.avc.decoder [1280x720]`）但 `inputFps=0, outputFps=0` —— 解码器无帧数据流入。
- **根因分析**：MediaTek 芯片通常仅支持 2-4 个并发 H.264 解码器。Feed PlayerPool（4 个）+ 独立页（1 个）= 5 个 → 超限；IndexedStack 中非活跃 controller 仅 pause 未 dispose，仍占用解码器槽位和 GPU 缓冲区。
- **解决方案**：独立播放页 `dispose()` 中立即 `pause()` + `dispose()` 释放硬件解码器；重试前先 dispose 旧 controller；监听 `controller.value.hasError` 及时反馈错误状态。

### 7.5 预加载与当前播放的带宽竞争

- **问题背景**：滑动到新视频时预加载同时进行，可能抢占当前视频的网络资源。
- **问题现象**：视频首帧加载慢，需要等待 1-2 秒才能看到画面。
- **根因分析**：预加载与当前视频的 `initialize()` 同时从网络下载数据，竞争同一 WiFi 通道。
- **解决方案**：`VideoPreloadManager` 增加 800ms 启动延迟，串行执行（`maxConcurrent = 1`），WiFi 感知（蜂窝网络自动暂停）。

### 7.6 商品讲解片段 seek 超出视频时长

- **问题背景**：商品有 `highlight_time` 定义讲解时间点，但演示视频仅 7 秒，种子数据设置为 5-35 秒。
- **问题现象**：seek 到超出视频时长的位置，无画面或跳转失败。
- **解决方案**：修正种子数据 highlight_time 为 1-6 秒范围；客户端 seek 增加 clamp 保护：`seekTo.clamp(0, (durSec - 0.5).ceil().clamp(0, durSec))`。

// ============ PlayerPool.acquire ============
Future<VideoPlayerController> acquire(String videoId, String url) async {
  // 1. videoId 命中 → refCount++ → 直接返回
  if (_players.containsKey(videoId)) {
    _players[videoId]!.refCount++;
    return _players[videoId]!.controller;
  }
  // 2. URL 去重（同一 URL 共享一个 ExoPlayer）
  for (final entry in _players.entries) {
    if (entry.value.url == url) {
      entry.value.refCount++;
      _players[videoId] = entry.value;  // 别名指向
      return entry.value.controller;
    }
  }
  // 3. 池满 → 淘汰 refCount 最小的（LRU 变体）
  _evictIfNeeded();
  // 4. 创建新 controller
  final controller = VideoPlayerController.networkUrl(Uri.parse(url));
  await controller.initialize();
  await controller.setLooping(true);
  _players[videoId] = _PooledPlayer(controller, 1, url);
  return controller;
}

// ============ VideoPreloadManager._processQueue ============
void _processQueue() {
  if (!canPreload) return;  // WiFi 感知
  while (_loadingCount < maxConcurrent && _queue.isNotEmpty) {

    _loadingCount++;
    _preloadOne(task).whenComplete(() {
      _loadingCount--;
      _processQueue();  // 串行：完成一个再处理下一个
    });
  }
}

## 9. 异常与边界处理

| 场景 | 处理方式 |
|------|----------|
| 视频 URL 为空 | `_initVideo` 检测空 URL → 设 `_videoError = true` → 显示错误 UI + 重试按钮 |
| 网络请求超时 | Dio `connectTimeout: 10s`, `receiveTimeout: 10s`；独立页 API 10s 超时 + 初始化 20s 超时 |
| WebSocket 断线 | 指数退避重连（1s→2s→4s→8s），最多 5 次 |
| API 返回 404/null data | `video_api.dart` 主动抛 `ClientException` |
| TypeError 未被捕获 | `catch(e)` 代替 `on Exception` |
| PageView 快速滑动 | `isActive` 标志控制：非当前视频 pause + release |
| ExoPlayer 实例超限 | 独立页 dispose 时立即 `pause()` + `dispose()` 释放 |
| 商品 seek 超时 | `clamp(0, durSec - 0.5s)` 边界保护 |
| 弹幕过多 | 最多同时 12 条，超出移除最旧，动画结束释放 AnimationController |
| 收藏商品数据过期 | 优先从 API 拉取最新商品数据，API 失败时 fallback 到本地存储 |

## 10. 总结

本人在项目中负责了**视频流浏览**和**直播间实时互动**两大核心模块，涵盖 FeedPage 滑动切换、VideoPlayerWidget 播放器组件、PlayerPool 播放器池、VideoPreloadManager 预加载、DanmakuOverlay 弹幕系统、WebSocket 实时通信、PiP 小窗播放、商品讲解片段定位、收藏页独立播放等子模块。

核心技术成果：
- **播放器池（PlayerPool）**：通过引用计数 + URL 去重 + LRU 淘汰，实现视频秒切和内存复用
- **优先级预加载**：WiFi 感知、串行执行、启动延迟，在不影响当前视频的前提下提前准备后续播放器
- **独立弹幕动画层**：AnimationController 驱动，限制并发数，动画结束立即释放
- **全局 PiP 状态管理**：跨页面保存/转移/复用 VideoPlayerController，实现无缝小窗播放
- **收藏页独立播放**：完全解耦的单视频播放页，不干扰 Feed 页的播放器池和预加载逻辑

调试过程中特别加深了对以下问题的理解：
- Dart `Error` 与 `Exception` 的类型体系差异
- Android ExoPlayer 硬件解码器的实例限制与资源管理
- Flutter 页面路由与 IndexedStack 生命周期对 platform view 的影响
- WebSocket URL 与 REST API URL 的路径独立性
